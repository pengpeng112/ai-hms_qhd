# AI-HMS 本地开发、真实环境模拟与内网 Linux 迁移指南
> 适用项目目录：`ai-hms_qhd`  
> 包含：`ai-hms-backend`（Go/Gin + PostgreSQL）与 `ai-hms-frontend`（React + Vite）
---
## 1. 代码结构与现状解析
## 1.1 后端（`ai-hms-backend`）
- 技术栈：Go 1.24.x、Gin、GORM、PostgreSQL
- 入口：`cmd/server/main.go`
- 配置：`config/config.go`（从 `.env` 读取）
- 数据库初始化：`internal/database/database.go`
- 自动迁移：`internal/database/migrate.go`
- 路由：`internal/api/v1/*`
- 外部集成：`internal/integrations/hdis/*`
关键行为（你后续开发要特别注意）：
1. 服务启动时即尝试连库，连不上也会继续启动（仅打印 Warning）。
2. `GIN_MODE=debug` 时会执行 `AutoMigrate`，`GIN_MODE=release` 时不会执行自动迁移。
3. 当前登录是硬编码账号：`admin / admin123`（仅演示用）。
4. 健康检查路由是 `/health`（不是 `/api/v1/health`）。
## 1.2 前端（`ai-hms-frontend`）
- 技术栈：React 19 + TypeScript + Vite + Antd
- 启动脚本：`npm run dev`
- API 客户端：`src/services/restClient.ts`
- 登录页：`src/pages/Login.tsx`
关键行为：
1. 前端请求基地址使用 `VITE_API_BASE_URL`。
2. `.env` 里有 `VITE_BACKEND_API_URL`，但代码当前**没有使用**该变量。
3. 因此开发时必须确保 `VITE_API_BASE_URL` 正确，否则会连错后端。

## 2. 本地开发环境（Windows）搭建步骤
## 2.1 准备依赖
- Go：建议 1.24.x（与 `go.mod` 对齐）
- Node.js：建议 20 LTS+
- npm：建议 10+
- PostgreSQL：15+
## 2.2 初始化数据库
在 PostgreSQL 中创建数据库（示例）：
```sql
CREATE DATABASE ai_hms_db;
```
如需独立账号（推荐）：
```sql
CREATE USER ai_hms WITH PASSWORD 'StrongPasswordHere';
GRANT ALL PRIVILEGES ON DATABASE ai_hms_db TO ai_hms;
```
## 2.3 配置后端环境变量
在 `ai-hms-backend` 目录下：
1. 复制 `.env.example` 为 `.env`
2. 最少修改这些项：
   - `DB_HOST`
   - `DB_PORT`
   - `DB_USER`
   - `DB_PASSWORD`
   - `DB_NAME`
   - `JWT_SECRET`
   - `APP_SECRET`
   - `CORS_ALLOWED_ORIGINS`（要包含前端地址）
开发建议：
- 首次本地联调时用 `GIN_MODE=debug`（便于自动迁移+日志排查）
## 2.4 启动后端
```bash
cd ai-hms-backend
go mod download
go run cmd/server/main.go
```
验证：
```bash
curl http://localhost:8080/health
```
## 2.5 导入演示数据（可选）
项目有 `scripts/test_data.sql`，可用于快速生成测试患者。  
注意：`import_test_data.sh` 里写死了数据库用户 `elliotxin`，你本地大概率需要改用户名或直接手动执行 SQL。
手动方式示例：
```bash
psql -h localhost -U ai_hms -d ai_hms_db -f scripts/test_data.sql
```
## 2.6 配置并启动前端
在 `ai-hms-frontend` 目录：
1. 确认 `.env.development` 含：
```env
VITE_API_BASE_URL=http://localhost:8080
```
2. 安装并运行：
```bash
cd ai-hms-frontend
npm ci
npm run dev
```
3. 打开：`http://localhost:5173`
登录测试账号：`admin / admin123`

## 3. 本地模拟“接近生产”的运行方式
目标：在开发机提前暴露部署风险，减少迁移到内网 Linux 时的返工。
## 3.1 建议分两个本地环境
1. **dev 环境**（日常开发）
   - 后端：`GIN_MODE=debug`
   - 自动迁移开启
   - 前端：`npm run dev`
2. **staging-like 环境**（准生产模拟）
   - 后端：`GIN_MODE=release`
   - 前端：`npm run build` + 静态托管
   - 使用“接近真实”的域名、端口、CORS、反向代理策略
## 3.2 重点核对项（模拟时必做）
- [ ] 前端是否只通过 `VITE_API_BASE_URL` 访问 API
- [ ] CORS 是否允许前端真实访问源
- [ ] 登录、鉴权、401 跳转逻辑是否正常
- [ ] 数据库迁移策略是否明确（release 不自动迁移）
- [ ] 日志是否能按 `app/error` 区分并可追踪
- [ ] HDIS 对接在“未配置/配置错误/令牌过期”时的提示是否可用
---
## 4. 迁移到内网 Linux 的推荐步骤
## 4.1 后端部署（systemd）
建议目录结构（参考你现有工作流）：
```text
/opt/ai-hms-backend/
  ├─ bin/
  ├─ config/
  ├─ logs/
  └─ backups/
```
构建 Linux 二进制（在构建机执行）：
```bash
cd ai-hms-backend
CGO_ENABLED=0 GOOS=linux GOARCH=amd64 go build -o ai-hms-server ./cmd/server
```
上传后：
- 放入 `/opt/ai-hms-backend/bin/`
- 准备 `/opt/ai-hms-backend/config/.env`
- 配置并启用 systemd 服务
## 4.2 前端部署（推荐 Nginx）
```bash
cd ai-hms-frontend
npm ci
npm run build
```
将 `dist/` 部署到 Nginx 静态目录。  
内网部署前，务必把 `VITE_API_BASE_URL` 改成内网后端地址（构建时生效）。
## 4.3 数据库迁移策略（重点）
由于后端在 `release` 不执行 `AutoMigrate`，建议你选一种固定方案：
1. **方案 A（快速）**：上线前临时用 debug 启动一次完成迁移，再切回 release。
2. **方案 B（推荐）**：引入明确的 SQL 迁移工具（如 goose/migrate），将表结构变更纳入版本化流程。
## 4.4 内网上线后验证清单
- [ ] `curl http://<backend-host>:8080/health` 返回成功
- [ ] 前端可正常登录
- [ ] 关键页面（患者列表/详情/排班/治疗）可读写
- [ ] 日志路径与权限正确
- [ ] 数据库连接池、磁盘、CPU、内存监控可见

3. 增加备份策略（数据库+配置+日志轮转）
4. 增加审计日志与告警（登录失败、同步失败、5xx）

## 7. 已发现的关键风险（建议尽快处理）
1. `Dockerfile` 基础镜像是 `golang:1.21-alpine`，但项目要求 Go 1.24，可能导致构建失败或行为不一致。
2. 前端 `.env` 中 `VITE_BACKEND_API_URL` 与代码读取变量不一致，易引发“本地能跑、部署后接口错地址”的问题。
3. 登录为硬编码账户，仅适合演示，不可直接用于正式环境。
4. release 模式不自动迁移数据库，若无独立迁移流程，发版易出现表结构不一致。
