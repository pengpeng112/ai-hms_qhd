# 患者管理与透析执行全流程测试计划

> 适用仓库：`ai-hms_qhd`
>
> 范围：患者管理、患者建档/查询/修改/删除限制、透析执行全流程、字典调用、数据库落库状态、代码审查。
>
> 执行原则：先只读核查，再使用测试患者做写入验证；不要修改真实患者数据；不要执行 DDL；不要启用 AutoMigrate；所有写入需记录测试患者 ID、治疗记录 ID、执行时间、请求 payload、数据库前后状态。

## 1. 测试目标

1. 验证患者管理页面从字典加载、患者列表查询、过滤、统计、新增建档、详情跳转、删除限制的完整链路。
2. 验证透析执行页面从患者列表、今日治疗记录创建、透前评估、当日处方读取、双人核对、医嘱、透中监测、透后评估、健康宣教、透析小结的完整链路。
3. 验证所有写入接口在旧血透库中的表、字段、状态码、时间字段、软删除/封存策略是否符合当前代码实现。
4. 验证字典类型兼容逻辑：前端统一字典 code 与后端 legacy Type 的映射是否完整，页面展示是否能正确从字典 code/name 回显。
5. 通过代码审查确认前端 UI 优化没有改变原 API、字段、业务状态流转和保存逻辑。
6. 将不确定字段、旧库枚举、无法验证项记录到“不确定项清单”，避免猜测。

## 4. 接口清单与预期数据库状态

### 4.1 患者管理接口

| 场景 | 前端调用 | 后端路由 | 主要旧库表 | 预期状态 |
|---|---|---|---|---|
| 患者列表 | `GET /api/v1/patients` | `patients.GET("")` | `Register_PatientInfomation`、`Register_OutCome`、`Plan_PatientPlan`、`Register_Diagnosis` | 返回分页、按 `TenantId=3` 过滤；在科/转出通过最新 `Register_OutCome.Type` 判断 |
| 患者统计 | `GET /api/v1/patients/stats` | `patients.GET("/stats")` | `Register_PatientInfomation`、`Register_OutCome` | 总数、在科、门诊、住院数量正确 |
| 新增患者 | `POST /api/v1/patients` | `patients.POST("")` | `Register_PatientInfomation`、`Register_Hospitalization`、`Register_IDInfomation`、`Register_FamilyMember` | 主表建档；扩展表按 payload 可选字段写入 |
| 患者详情 | `GET /api/v1/patients/:id` | `patients.GET("/:id")` | `Register_PatientInfomation` | 返回单患者核心字段 |
| 基础档案读取 | `GET /api/v1/patients/:id/basic-info` | `patients.GET("/:id/basic-info")` | 多个 `Register_*` 表 | 返回个人、医疗、联系方式、社保/证件信息 |
| 基础档案更新 | `PUT /api/v1/patients/:id/basic-info` | `patients.PUT("/:id/basic-info")` | 多个 `Register_*` 表 | 只更新映射字段，不新增未知字段 |
| 患者更新 | `PUT /api/v1/patients/:id` | `patients.PUT("/:id")` | `Register_PatientInfomation` | 当前服务仅明确更新 `PatientType`、`ExpenseType`、`TreatmentStatus` |
| 患者删除 | `DELETE /api/v1/patients/:id` | `patients.DELETE("/:id")` | 无写入 | 当前服务返回“老库患者暂不支持直接删除”；前端应提示失败，不应物理删除 |

### 4.2 透析执行接口

| 场景 | 前端调用 | 后端路由 | 主要旧库表 | 预期状态 |
|---|---|---|---|---|
| 读取今日治疗 | `GET /api/v1/patients/:id/treatment?date=YYYY-MM-DD` | `r.GET("/patients/:id/treatment")` | `Treatment_Treatment` 及子表 | 按 `DATE(COALESCE(StartTime, SignInTime, ReceptionTime, CreateTime))` 查询最新记录 |
| 创建今日治疗 | `POST /api/v1/treatments` | `treatments.POST("")` | `Treatment_Treatment` | 写入 `PatientId`、`TenantId`、`Status`、`CreateTime`、`LastModifyTime`、`NurseSummary/TreatmentSummary` |
| 更新治疗状态 | `PUT /api/v1/treatments/:id` 或 `/:id/status` | `Update` / `UpdateStatus` | `Treatment_Treatment` | 上机状态补 `StartTime`；完成状态补 `EndTime` |
| 透前评估 | `PUT /api/v1/treatments/:id/before-signs` | `SaveBeforeSigns` | `Treatment_BeforeSigns`、`Treatment_BeforeSymptom`、JSON 快照 | upsert 生命体征；症状项替换；保存 JSON 快照 |
| 首次核对 | `PUT /api/v1/treatments/:id/first-check` | `SaveFirstCheck` | `Treatment_BeforeCheck`、`Treatment_Action` | upsert `Treatment_BeforeCheck`，创建/更新 action code `70` |
| 二次核对 | `PUT /api/v1/treatments/:id/second-check` | `SaveSecondCheck` | `Treatment_Action`、JSON 快照 | 创建/更新 action 和 JSON 快照；不一定写 `Treatment_BeforeCheck` |
| 消毒登记 | `PUT /api/v1/treatments/:id/disinfection` | `SaveDisinfection` | 需执行时核查 | 验证对应字段/JSON/Action 是否落库，代码审查确认 |
| 透中监测新增 | `POST /api/v1/treatments/:id/during-params` | `CreateDuringParam` | `Treatment_DuringParam`、可能同步 `Treatment_DuringSigns` | 新增监测点，返回新 ID |
| 透中监测修改 | `PUT /api/v1/treatments/:id/during-params/:paramId` | `UpdateDuringParam` | `Treatment_DuringParam` | 指定记录更新，`LastModifyTime` 更新 |
| 透中监测删除 | `DELETE /api/v1/treatments/:id/during-params/:paramId` | `DeleteDuringParam` | `Treatment_DuringParam` | 需确认是物理删除还是状态标记 |
| 透后评估保存 | `PUT /api/v1/treatments/:id/after-signs` | `SaveAfterSigns` | `Treatment_AfterSigns`、`Treatment_AfterSymptom`、`Treatment_Treatment`、JSON 快照 | upsert 透后体征、症状、治疗小结相关字段 |
| 透后提交 | `PUT /api/v1/treatments/:id/post-assessment-submit` | `SubmitPostAssessment` | `Treatment_AfterSigns`、`Treatment_Treatment` | 保存透后并将治疗状态推进到完成；`EndTime` 应存在 |
| 透析小结 | `PUT /api/v1/treatments/:id/summary` | `SaveSummary` | `Treatment_Treatment` | 更新 `TreatmentSummary`/`NurseSummary` 或相关总结字段 |
| 健康宣教 | `POST /api/v1/patients/:id/health-educations` 或实际路由 | `HealthEducationHandler` | `Auxiliary_PatientHealthEducation` | 写入宣教内容、结果、签名、完成时间 |

## 5. 字典调用测试计划

### 5.1 字典类型重点

| 前端统一 code | 旧库 Type | 使用位置 | 预期 |
|---|---|---|---|
| `INSURANCE_TYPE` | `ExpenseType` | 患者列表、建档、基础档案 | 能展示医保名称；级联医保能加载 |
| `PATIENT_TYPE` | `PatientType` | 患者列表、建档、统计 | 门诊/住院等能展示 |
| `DIALYSIS_MODE` | `DialysisMethod` | 患者列表、建档、方案、处方 | `HD/HDF/HP` 等能保存和回显 |
| `ID_TYPE` | `IDType` | 建档、基础档案 | 身份证等能校验与保存 |
| `VISIT_CATEGORY` | `HospPatientType` | 建档、基础档案 | 门诊/住院就诊类别可选 |
| `BLOOD_TYPE_ABO` | `ABOType` | 基础档案 | ABO 血型可回显 |
| `BLOOD_TYPE_RH` | `RHType` | 基础档案 | Rh 血型可回显 |
| `VASCULAR_ACCESS` | `AccessType` | 治疗方案/血管通路 | 通路类型可回显 |
| `VASCULAR_SITE` | `AccessPosition` | 治疗方案/血管通路 | 通路部位可回显 |

### 5.2 字典接口验证

1. 调用 `GET /api/v1/dict/types`，确认返回 legacy 与统一 code 映射后的类型。
2. 调用 `GET /api/v1/dict/items/INSURANCE_TYPE?isEnabled=true`，确认可返回 `ExpenseType` 数据。
3. 调用 `GET /api/v1/dict/items/INSURANCE_TYPE/tree?isEnabled=true`，确认建档医保 Cascader 有数据。
4. 调用 `GET /api/v1/dict/items/DIALYSIS_MODE?isEnabled=true`，确认包含当前旧库透析方式。
5. 如果接口为空，直接查 `CodeDictionary_CodeDictionarys`：

```sql
SELECT "Type", "Code", "Name", "IsDisabled"
FROM "CodeDictionary_CodeDictionarys"
WHERE "Type" IN ('ExpenseType', 'PatientType', 'DialysisMethod', 'IDType', 'HospPatientType')
ORDER BY "Type", "Sort", "Id";
```

6. 若旧库有数据但 API 为空，记录为“后端字典映射/过滤问题”。
7. 若 API 有数据但页面为空，记录为“前端字典 code/缓存/回显问题”。

## 6. 患者管理端到端用例

### 6.1 用例 P-001：患者列表初始加载

| 步骤 | 操作 | 预期 |
|---|---|---|
| 1 | 打开患者管理页面 | 默认筛选为“在科患者” |
| 2 | 观察 Network | 请求 `GET /api/v1/patients?page=1&pageSize=50&onlyActive=true` |
| 3 | 观察统计 | 请求 `GET /api/v1/patients/stats` |
| 4 | 检查页面 | 患者姓名、性别、年龄、医保、类型、透析模式正常展示，无乱码/undefined |
| 5 | 数据库核对 | 列表患者应来自 `Register_PatientInfomation` 且最新 `Register_OutCome.Type='10'` |

参考 SQL：

```sql
WITH latest_outcome AS (
  SELECT DISTINCT ON ("PatientId") "PatientId", "Type", "OutComeTime", "CreateTime"
  FROM "Register_OutCome"
  WHERE "TenantId" = 3
  ORDER BY "PatientId", "OutComeTime" DESC, "CreateTime" DESC
)
SELECT p."Id", p."Name", p."PatientType", p."ExpenseType", oc."Type"
FROM "Register_PatientInfomation" p
JOIN latest_outcome oc ON oc."PatientId" = p."Id" AND oc."Type" = '10'
WHERE p."TenantId" = 3
ORDER BY p."CreateTime" DESC
LIMIT 20;
```

## 7. 透析执行端到端用例

### 7.1 用例 T-001：执行页加载患者与今日治疗

| 步骤 | 操作 | 预期 |
|---|---|---|
| 1 | 打开透析执行页面 | 请求 `GET /api/v1/patients?page=1&pageSize=200` |
| 2 | 选择测试患者 | 请求 `GET /api/v1/patients/:id/treatment?date=YYYY-MM-DD` |
| 3 | 若无记录 | 页面显示“暂无治疗记录”，可创建 |
| 4 | 若接口 404 | 前端应进入 `missing` 状态，不弹阻断错误 |

## 10. 数据库核查注意事项

1. 所有旧库表名和列名大小写敏感，SQL 必须使用双引号。
2. 所有写入前先记录旧状态，写入后再查新状态。
3. 不要对生产/真实患者执行删除、封存、完成治疗等不可逆状态操作。
4. 如果必须清理测试数据，先确认后端是否支持软删除；当前患者删除明确不支持，不能直接物理删旧库数据。
5. 不要执行 `UPDATE/DELETE` 清理真实库；如需要清理测试记录，先提交清理方案给负责人确认。
