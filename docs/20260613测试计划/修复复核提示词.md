## 需要复核的 8 项修复

### FIX-1（严重）：DialysisExecution 重复创建治疗记录竞态

**文件**：`ai-hms-frontend/src/pages/dialysis-processing/DialysisExecution.tsx`

**问题**：`ensureTodayTreatment` 在 `currentTreatment === null` 时调用 `restApi.createTreatment()`。用户双击"创建治疗记录"按钮或快速切换 tab 触发两次保存时，两个并发调用都读到 `currentTreatment === null`，各自创建治疗记录，产生重复。

**修复**：在 `ensureTodayTreatment` 入口增加 `ensuringRef`（useRef）互斥锁，并发调用时第二个直接返回 null。

**复核要点**：
1. `ensuringRef` 是否在 `finally` 中正确释放（不会死锁）。
2. 是否与已有的 `treatmentRequestIdRef` 竞态保护产生冲突。
3. `handleCreateTodayTreatment` 按钮是否也需要加 `disabled` 守卫；如果确认双击仍会造成重复提示、重复请求或静默失败，做最小修复。
4. 并发调用直接返回 null 是否会导致保存按钮无反馈；如果调用方已有保存提示或 loading 机制，说明现状可接受。

### FIX-2（高）：ensureTodayTreatment 缺少 try/catch

**文件**：同 FIX-1

**问题**：`restApi.updateTreatment` 和 `restApi.createTreatment` 抛出异常时无 catch，错误作为未处理 Promise rejection 传播，用户无任何反馈。

**修复**：整个 `ensureTodayTreatment` 包裹 try/catch，catch 中 `message.error(getErrorMessage(error))` 并 return null。

**复核要点**：
1. catch 中的 `getErrorMessage` 是否已正确 import（检查文件头 import）。
2. 调用 `ensureTodayTreatment` 的 handler 后续子表保存失败时是否仍可能无提示；这不是 FIX-2 的直接问题，除非会导致未处理 Promise rejection。

### FIX-3（高）：PatientList 删除后统计卡片不刷新

**文件**：`ai-hms-frontend/src/pages/PatientList.tsx`

**问题**：删除患者成功后只调了 `loadPatients()`，未递增 `refreshKey`，导致顶部统计卡片（总数/在科/门诊/住院）显示旧值。

**修复**：在 `handleDeletePatient` 的 `onOk` 回调中，`loadPatients()` 之后增加 `setRefreshKey(k => k + 1)`。

**复核要点**：
1. `refreshKey` 变化是否会触发 `patientStats` effect 重新加载。
2. 注意后端 `Delete` 当前返回 `"老库患者暂不支持直接删除"` 错误，所以删除实际会走 catch 分支。此修复是预防性的——一旦后端支持删除，统计会正确刷新。
3. `loadPatients()` 当前未 `await`；如果删除成功后立即 `setRefreshKey` 会造成重复加载但不影响正确性。除非确认有可见竞态，否则不要扩大修改。

### FIX-4（高）：PatientList 分页空页面陷阱

**文件**：同 FIX-3

**问题**：分页栏的显示条件是 `!loading && filteredPatients.length > 0`。当翻到末页恰好 0 条结果时，分页栏消失，用户无法点"上一页"返回。同时"下一页"按钮的 `disabled` 条件是 `patients.length < pageSize`，不准确。

**修复**：
1. 分页栏显示条件改为 `!loading`（去掉 `filteredPatients.length > 0`）。
2. "下一页" `disabled` 改为 `currentPage * pageSize >= (total || patients.length)`。

**复核要点**：
1. 当 `total` 为 0 时，`currentPage * pageSize >= 0` 恒为 true，"下一页"被禁用——是否正确。
2. 客户端过滤（today/mine）后 `filteredPatients.length` 可能小于 `patients.length`，此时 total 仍为服务器端总数，分页计数是否误导用户。
3. 如果当前页超过最大页且服务端返回空列表，分页栏是否仍允许用户返回上一页。

### FIX-5（高）：equipment.ts 四个查询函数缺少 success 检查

**文件**：`ai-hms-frontend/src/services/equipment.ts`

**问题**：`getEquipmentById`、`getEquipmentDisinfections`、`getEquipmentUsageLogs`、`getEquipmentMaintenanceRecords` 直接访问 `response.data.data`，未先检查 `response.data.success`。后端返回错误时 `data` 为 undefined，`toEquipmentInfo(undefined)` 会崩溃。

**修复**：四个函数均在访问 `.data` 前增加 `if (!response.data.success) throw new Error('...')`。

**复核要点**：
1. 本次提到的四个函数是否都在访问 `.data` 前检查 `response.data.success`。
2. 是否还有同文件中同类高风险读取遗漏 success 检查（例如 `getEquipmentList`）。如果遗漏且修复很小，可以一并修复；不要重构整个 service。
3. 错误消息是否为中文且与函数用途匹配。

### FIX-6（高）：dictApi 变更操作后不清除 DictCache

**文件**：`ai-hms-frontend/src/services/dictApi.ts`

**问题**：`createItem`、`updateItem`、`deleteItem`、`toggleItemEnabled` 执行后不清除 `DictCache`，导致下拉框在缓存 TTL（5分钟）内显示旧数据。

**修复**：
- `createItem`：成功后 `dictCache.clear(data.typeCode)`
- `updateItem`：成功后 `dictCache.clear(data.typeCode)`（仅当 data 中有 typeCode）
- `deleteItem`：成功后 `dictCache.clear()`（全清，因为 id 无法反查 typeCode）
- `toggleItemEnabled`：成功后 `dictCache.clear()`（同上）

**复核要点**：
1. `dictCache.clear()` 是否在 `post`/`put`/`del` 调用之后执行（不会阻塞请求本身）。
2. `updateItem` 的 `data.typeCode` 是否一定存在；如果调用方可能只传 `{ name: 'xxx' }` 不传 typeCode，则缓存不会清除。若确认存在这种调用方式，优先改为更新成功后 `dictCache.clear()` 全清，避免旧字典显示。
3. `del()` 函数在 204 No Content 时返回 `undefined as T`——`await del<void>(...)` 的返回值是否被正确忽略。

### FIX-7（中）：patient_service / treatment_service pageSize 无上限

**文件**：
- `ai-hms-backend/internal/services/patient_service.go`（List 函数）
- `ai-hms-backend/internal/services/treatment_service.go`（List 函数）

**问题**：`req.PageSize` 仅检查 `<= 0` 时设为 20，无上限。请求 `?pageSize=999999999` 可导致数据库扫描和内存暴涨（DoS 风险）。

**修复**：在 `if req.PageSize <= 0 { req.PageSize = 20 }` 之后增加 `if req.PageSize > 200 { req.PageSize = 200 }`。

**复核要点**：
1. `patient_service.go` 和 `treatment_service.go` 的 `List` 是否都把 `PageSize` 限制到 200。
2. 是否有调用方依赖大于 200 的 pageSize；如果没有证据，不要扩大兼容。
3. 其他 service 的 pageSize 上限属于范围外风险，只在报告里列出，不要本次批量修改。

### FIX-8（高）：DEV_SUPERUSER 在 release 模式下仍可绕过鉴权

**文件**：`ai-hms-backend/internal/smart_schedule/api/api.go`

**问题**：`var devSuperuser = strings.EqualFold(os.Getenv("DEV_SUPERUSER"), "true")` 不区分 Gin 运行模式。如果生产环境误设 `DEV_SUPERUSER=true`，所有角色校验被绕过。

**当前修复**：改为 `var devSuperuser = strings.EqualFold(os.Getenv("DEV_SUPERUSER"), "true") && gin.Mode() != gin.ReleaseMode`。

**已知高风险点**：当前写法很可能仍不可靠。`api.go` 的包级变量会在 `main()` 执行前初始化，而 `cmd/server/main.go` 是在 `main()` 第 33 行才调用 `gin.SetMode(cfg.Server.Mode)`。因此 `devSuperuser` 求值时 `gin.Mode()` 可能仍是 Gin 默认 debug 模式，导致生产 release 配置无法生效。

**复核要点**：
1. 优先确认上述初始化顺序问题是否成立。
2. 如果成立，直接最小修复：不要在包级变量中缓存 `gin.Mode()` 判断，改为运行时函数，例如 `isDevSuperuser()`，在 `guard` 和日志输出处动态判断 `strings.EqualFold(os.Getenv("DEV_SUPERUSER"), "true") && gin.Mode() != gin.ReleaseMode`。
3. 修复后确认 release 模式下即使 `DEV_SUPERUSER=true` 也不会放行未带角色的请求。
