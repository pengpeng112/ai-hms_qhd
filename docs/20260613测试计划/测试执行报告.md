## 3. 代码审查结果

### 3.1 后端患者服务 (`patient_service.go`)

| 等级 | 位置 | 问题 |
|---|---|---|
| **高** | `Update:957` | **缺少 TenantId 过滤**。`s.db.First(&patient, '"Id" = ?', id)` 没有 `"TenantId" = ?`，跨租户冲突风险。相比 `Get:640` 有正确过滤 |
| **高** | `Create:865 vs 888` | **TenantId 不一致**：`Register_Hospitalization` 使用参数 `tenantID`，`Register_IDInfomation`/`FamilyMember` 硬编码为 `LegacyTenantID`(3) |
| **中** | `Update:965-973` | **静默忽略 6/9 字段**：`UpdateRequest` 定义了 `BedNumber/Diagnosis/DryWeight/DefaultMode/DoctorID/DoctorName` 但完全不处理 |
| **中** | `Create:842` | `createBasicInfo` 中的 `Register_PatientInfomation` UPDATE `WHERE "Id" = ?` 无 TenantId 过滤 |
| **中** | `GetStats:598-608` | **活跃统计可能遗漏**：仅统计有 `Register_OutCome` 记录的患者，无 OutCome 记录的患者不会被计入活跃 |

## 7. 结论与建议

### 7.1 总体评估

系统整体可构建、可运行。患者管理和透析执行的核心链路完整，API 路径一致。发现 **4 个高优先级缺陷**（2 个 TenantId 过滤缺失、1 个 TenantId 不一致、1 个字典编码不一致）和 **8 个中优先级缺陷**（字段忽略、类型错误、事务缺失等）。

### 7.2 建议修复优先级

1. **（高）TenantId 过滤**：`patient_service.go Update` + `treatment_service.go SaveSummary` 补 `TenantId` 条件
2. **（高）TenantId 一致性**：`patient_service.go Create` 子表统一使用 `LegacyTenantID`；`treatment_service.go Create` 改为 `LegacyTenantID`
3. **（高）字典编码**：`dict_service.go ListTypes` 返回 unified Code 或前端做 legacy→unified 转换
4. **（中）类型修复**：`SaveDisinfection` StartTime/EndTime 改为 time.Time 类型
5. **（中）事务包裹**：多表写入方法加数据库事务
6. **（中）前端 Stats**：补 success 检查
