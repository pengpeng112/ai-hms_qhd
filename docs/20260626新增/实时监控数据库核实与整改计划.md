# 实时监控数据库核实与整改计划

核实时间：2026-06-26

核实方式：读取 `ai-hms-backend/.env` 后仅执行 PostgreSQL 只读查询，未输出任何连接密码，未执行 DDL/DML。

## 一、核实结论

### 1. 已确认存在的老库字段

- `Plan_PatientPrescription."DialysisDuration"`：存在，`numeric`。
- `Treatment_BeforeSigns."Weight"`：存在，`numeric`。
- `Device_DMLog."VenousPressure"`：存在，`numeric`。
- `Device_DMLog."ArterialPressure"`：存在，`numeric`。
- `Device_DMLog."TMP"`：存在，`numeric`。
- `Device_DMLog."BF"`：存在，`numeric`。
- `Device_DMLog."Conductivity"`：存在，`numeric`。
- `Device_DMLog."UFVolume"`：存在，`numeric`。
- `Device_DMLog."UFSetVolume"`：存在，`numeric`。
- `Device_DMLog."TreatmentTime"`：存在，`numeric`。
- `Register_VascularAccess."AccessType"` / `"IsDefault"` / `"OperationTime"` / `"IsDisabled"`：存在。

### 2. 当前代码引用但数据库不存在的对象

- `lab_reports.patient_id`：不存在。
- `lab_report_items.lab_report_id` / `item_name` / `result_value` / `tested_at`：不存在。
- `dry_weight_assessment.tenant_id` / `patient_id` / `rna_setting` / `created_at`：不存在。

影响：当前 `monitoring_service.go` 中 RNa 完成率的 C_pre 与目标 RNa 查询会取不到数据，并会对不存在表发起 SQL。由于代码未检查这些 `Raw(...).Scan(...)` 的错误，接口可能仍返回，但 RNa 会长期 `available=false`，同时可能产生日志噪音或隐藏数据问题。

### 3. 可替代的数据来源

- C_pre 血清钠可从老库原生 LIS 表读取：
  - `LIS_Examination`：有 `TenantId`、`PatientId`、`ResultTime`。
  - `LIS_ExaminationItem`：有 `TenantId`、`ExaminationId`、`ItemName`、`ItemCode`、`Result`、`Unit`。
  - 核实到 `TenantId=3` 下存在 4775 条钠项目，样例为 `ItemName=钠`、`ItemCode=NA`、`Result=138`、`Unit=mmol/L`。
- 仓库已有新表 `exam_reports` / `exam_report_items`，但本库钠项目数量为 0，且字段名是 `item_result`、时间在 `exam_reports.exam_date`，不是当前代码里的 `lab_report_items.result_value/tested_at`。
- `external_patient_mappings` 存在，可作为外部患者与 legacy 患者映射参考。
- `Treatment_NA_Memo` 存在 `Na_s_pre`、`Na_d`、`Na_target`、`Na_actual`、`R_Na` 等字段，但本次核实 `R_Na` 全部为空，且该表未显示 `TenantId`，若使用必须通过 `TreatmentId` 关联 `Treatment_Treatment` 加租户边界。

### 4. 单位核实

- `Plan_PatientPrescription."DialysisDuration"` 单位是小时，不是分钟。
  - 抽样范围：`1..8`，平均约 `3.96`。
  - 当前代码把它当分钟使用，会导致 `plannedEnd`、倒计时、UFR 计算错误。
- `Device_DMLog."TreatmentTime"` 单位是分钟。
  - 抽样中位数约 `120`，P95 约 `230`。
- `Device_DMLog."UFVolume"` / `"UFSetVolume"` 单位是 ml，不是 L。
  - 抽样样例：`UFVolume=390`、`UFSetVolume=2000`。
  - 当前代码把 `UFSetVolume` 当 L 后再乘 `1000` 计算 UFR，会放大约 1000 倍。
  - 当前 RNa 完成率也会把 `UFVolume` ml 当 L 使用，结果错误。
- `Device_DMLog."Conductivity"` 是原始电导率，常见范围约 `13.8..14.4`。
  - 当前 `Conductivity * 9.9` 得到透析液钠约 `137..143 mmol/L`，这个换算口径可继续保留。
- `Register_VascularAccess."AccessType"` 实际值主要是中文长文本：
  - `自体动静脉内瘘AVF`
  - `无隧道和涤纶套的透析导管NCC`
  - `带隧道和涤纶套的透析导管TCC`
  - `移植物动静脉内瘘AVG`
  - 少量 `AVF`

### 5. 当前库数据状态

- 核实当天无活动治疗：`active_today=0`。
- 当天 `Device_DMLog` 只有 1 条，且时间为 2030 年、关键数值为空。
- 全量 `Device_DMLog` 有约 920 万行，2024 年以后有效数据充足，可用于单位判断，但无法验证今天实时页面的在线效果。

## 二、必须整改项

### P0-1 修正透析时长单位

文件：`ai-hms-backend/internal/services/monitoring_service.go`

要求：

- 后端对外 `EstimatedDuration` 继续输出分钟，保持前端现有逻辑不变。
- SQL 读到 `DialysisDuration` 后必须从小时转换为分钟。
- 建议新增小函数：

```go
func dialysisDurationMinutes(v float64) float64 {
    if v <= 0 {
        return 240
    }
    if v <= 24 {
        return v * 60
    }
    return v
}
```

- 在 `GetLiveData` 赋值 `EstimatedDuration` 前转换。
- 在 `GetTreatmentTrend` 计算 `PlannedEnd` 前转换。
- 测试覆盖：输入 `4` 应输出 `240`，输入 `240` 应保持 `240`。

验收：

- `plannedEnd = start + 4h`，不是 `start + 4min`。
- 前端倒计时显示约 `240min`。
- UFR 分母按小时正确计算。

### P0-2 修正 UF 单位

文件：`ai-hms-backend/internal/services/monitoring_service.go`

要求：

- 明确内部 RNa 和 UFR 计算使用 L。
- `Device_DMLog."UFVolume"` / `"UFSetVolume"` 从 ml 转 L 后再参与计算。
- 对外 JSON 字段 `ufVolume` / `ufGoal` 需要二选一：
  - 推荐：后端输出 L，前端继续按 L 展示，并在必要处标注单位。
  - 如果前端历史语义是 ml，则新增内部变量用于计算，不改变 JSON；但这会增加混乱，不推荐。
- 建议新增小函数：

```go
func mlToL(v float64) float64 {
    if v <= 0 {
        return 0
    }
    return v / 1000
}
```

- `d.UFGoal = mlToL(dev.UFSetVolume)`。
- `d.UFVolume = mlToL(dev.UFVolume)`。
- `evalAlarms` 中 UFR 继续使用 `d.UFGoal * 1000 / dryWeight / hours`，此时 `d.UFGoal` 已是 L。
- RNa 完成率传入 `UFActual` 时使用 L。

验收：

- `UFSetVolume=3000 ml`、`DryWeight=60 kg`、`Duration=4 h` 时，UFR 应为 `12.5 ml/kg/h`。
- 不得出现几千到几万的 UFR。

### P0-3 修正 C_pre 查询来源

文件：`ai-hms-backend/internal/services/monitoring_service.go`

当前错误来源：

- 代码查询 `lab_reports` / `lab_report_items`，本库不存在这些表。

整改要求：

- 改查老库 LIS：`LIS_Examination` + `LIS_ExaminationItem`。
- 必须带 `TenantId`。
- 以 `LIS_Examination.PatientId` 匹配 legacy 患者 ID。
- 最近一次钠结果按 `LIS_Examination.ResultTime DESC` 取。
- 结果字段用 `LIS_ExaminationItem.Result`。
- 钠项目条件建议：`ItemCode ILIKE 'NA' OR ItemName LIKE '%钠%'`。
- 保留现有数字清洗逻辑，兼容 `140↓`、`>145`、`140 mmol/L`。

推荐 SQL 口径：

```sql
SELECT DISTINCT ON (e."PatientId")
       e."PatientId",
       i."Result",
       e."ResultTime"
FROM "LIS_ExaminationItem" i
JOIN "LIS_Examination" e
  ON e."Id" = i."ExaminationId"
 AND e."TenantId" = i."TenantId"
WHERE e."TenantId" = ?
  AND e."PatientId" IN ?
  AND (i."ItemCode" ILIKE 'NA' OR i."ItemName" LIKE '%钠%')
ORDER BY e."PatientId", e."ResultTime" DESC;
```

验收：

- 能从当前库取到患者最近钠结果。
- 查询不再引用 `lab_reports` / `lab_report_items`。
- 查询必须有租户边界。

### P0-4 处理 RNa 目标来源缺失

当前问题：

- `dry_weight_assessment` 在当前数据库不存在。
- `Treatment_NA_Memo.R_Na` 存在但全为空，不能作为可靠目标 RNa。

整改建议：

- 短期：不要查询 `dry_weight_assessment`，避免不存在表错误；RNa 完成率在缺少目标 RNa 时明确返回 `Available=false`，前端显示“待处方”。
- 中期：确认 RNa 处方到底落库在哪张表。如果项目已有新 RNa 模块但未部署表，必须把部署 SQL 纳入 DBA 执行清单。
- 如果决定使用 `Treatment_NA_Memo`，必须先确认 `R_Na` 写入逻辑；当前库 `R_Na` 全为空，不能直接用。

验收：

- 接口不再查询不存在表。
- 缺少 RNa 目标时页面稳定显示待数据，不产生 SQL 错误。

## 三、建议整改项

### P1-1 对关键查询错误显式处理

当前多处 `Find` / `Raw(...).Scan(...)` 没有检查 `.Error`。

建议：

- 主查询失败应直接返回错误。
- 辅助数据查询失败可降级为空，但至少记录日志或返回可诊断信息。
- 对不存在表这种配置错误，不应静默吞掉。

### P1-2 限制床卡趋势点数量

当前 `GetLiveData` 会取所有活动治疗的整场 `Treatment_DuringSigns`。如果当天活动治疗多、记录密集，响应体会变大。

建议：

- 床卡只取最近若干点或按 stride 限制。
- 完整趋势继续走 `/monitoring/treatments/:id/trend`。

### P1-3 通路类型阈值归一化

实际 `AccessType` 是中文长文本，阈值引擎需要确认是否能识别：

- `自体动静脉内瘘AVF` 应归一到 AVF。
- `移植物动静脉内瘘AVG` 应归一到 AVG。
- `带隧道和涤纶套的透析导管TCC` 应归一到 catheter/TCC。
- `无隧道和涤纶套的透析导管NCC` 应归一到 catheter/NCC。

验收：

- VP 阈值分层不因中文长文本落入默认分支。

### P1-4 保持 IDH 为 stub

当前 IDH 真模型未完成 `.pkl` 解包、微服务部署和样本对拍。

要求：

- 不要把 `HTTPScorer` 注入生产路径。
- 不要让 IDH 风险参与三色告警。
- 后续真模型上线必须另起任务：模型文件、特征窗口、训练样本 `predict_proba` 对拍、误差阈值、服务超时降级。
