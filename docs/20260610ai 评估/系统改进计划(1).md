# AI-HMS 系统改进计划

> 制定日期：2026-06-10
> 配套文档：《系统评估报告.md》
> 原则：先稳生产库与安全红线，再治工程债与可维护性；每项改动**一次只动一个模块、保证可回滚**；前端 API 契约保持不变。

---

## 阶段总览

| 阶段 | 目标 | 关键项 | 建议周期 |
|---|---|---|---|
| 阶段一（立即） | 消除生产库 / 启动可用性高危 | P0-1、P0-2、P1-3 | 1 周内 |
| 阶段二（近期） | 安全加固与编码治理 | P1-1、P1-2、P1-4 | 2–3 周 |
| 阶段三（中期） | 工程债与可维护性 | P2-1 ~ P2-4 | 1–2 月 |

## 阶段一：立即（生产库 / 高危）

### 任务 1.1　消除运行时 DDL（P0-1）

**问题**：`smart_schedule/api/api.go:41` 在服务注册时调用 `repo.EnsureIndexes`，对老生产库执行 `CREATE UNIQUE INDEX`，违反"运行时严禁 DDL"红线，且可能因存量脏数据建索引失败导致启动失败。

**方案**：
1. 将 `repo.go:113-124` 的两条 `CREATE UNIQUE INDEX` 抽到外部 SQL 脚本（如 `scripts/schedule_unique_indexes.sql`），走 DBA 审核 + 手工执行流程（与既定守则一致）。
2. `api.go:41` 移除 `EnsureIndexes` 调用；改为**启动时只校验索引是否存在**（查 `pg_indexes`），缺失则告警日志 + 健康检查标红，**不自动创建**。
3. 执行外部 SQL 前先跑一次**重复数据探测**查询，确认 `uq_ps_patient_slot`/`uq_ps_machine_slot` 对应列无重复，避免建唯一索引失败。

**验收标准**：
- [ ] 代码路径中不存在任何 `CREATE/ALTER/DROP` DDL（`grep -rE "CREATE (TABLE|INDEX|UNIQUE)|ALTER TABLE|DROP " internal/` 仅命中测试文件）
- [ ] 服务在索引不存在时仍能启动并通过健康检查告警提示
- [ ] 外部 SQL 脚本含重复数据探测语句 + 回滚说明

**影响面**：backend（smart_schedule）

---

### 任务 1.2　核查老库并行写入保障（P0-2）

**问题**：应用与老系统并行直写同一生产库，缺并发与完整性保障会导致脏写。

**方案（核查 + 补齐）**：
1. **写操作盘点**：列出所有 `Create/Save/Update/Delete/Exec` 写路径，标注目标表与触发场景。
2. **审计**：确认 `RequestLogger` + 业务写操作是否落审计（谁/何时/改了哪条/前后值）。缺失的关键临床表（医嘱、透析记录、排班）补审计。
3. **并发保护**：关键表确认是否有唯一约束 + 乐观锁（`updated_at`/version）。上机/下机、排班确认等高并发操作必须有幂等键。
4. **灰度开关**：确认每个已迁移模块有独立开关，可快速切回老系统读路径。

**验收标准**：
- [ ] 产出《老库写操作清单》含表名/场景/保护手段
- [ ] 医嘱、透析记录、排班三类关键写操作均有审计与幂等/乐观锁
- [ ] 每个已上线模块有可回滚开关并验证过切换

**影响面**：backend、docs

---

### 任务 1.3　关闭演示账号与预览旁路（P1-3）

**方案**：
1. 确认 `admin / admin123` 在生产库不存在或已禁用；保留则改强密码 + 记录。
2. `restAuth.ts:98` 的 `local-preview-token` 旁路用编译期环境变量（`import.meta.env.DEV`）门控，确保 `vite build` 产物不含该路径。
3. 后端拒绝任何非真实签发的 token（核查 `local-preview-token` 不会被后端接受）。

**验收标准**：
- [ ] 生产构建产物 grep 不到 `local-preview-token` 可达路径
- [ ] 演示账号在生产环境登录失败或已强化
- [ ] 后端对伪造 token 返回 401

**影响面**：frontend、backend

## 阶段二：近期（安全 / 编码）

### 任务 2.1　权限门禁从角色名迁到权限码（P1-1）

**问题**：`main.go:171` 硬编码 `RequireRoles("ADMIN","管理员","安全管理员","运维管理员")`，脆弱且分散。

**方案**：
1. 复用现有 permission 模块，为 admin 类路由定义权限码（如 `system:manage`、`dict:write`、`user:manage`）。
2. 新增 `RequirePermission(code...)` 中间件，查 claims/DB 中用户权限码。
3. `main.go` 的 admin 组逐路由替换为权限码门禁，保留角色名仅作兼容兜底（可配置）。

**验收标准**：
- [ ] admin 路由门禁基于权限码而非角色字面量
- [ ] 中文角色名改动不影响权限判定（加测试用例）
- [ ] 权限码与角色映射集中可配

**影响面**：backend（middleware、main、permission_service）

---

### 任务 2.2　JWT 存储加固（P1-2）

**方案（择一，按风险偏好）**：
- **方案 A（推荐）**：accessToken 改 httpOnly + Secure + SameSite cookie，后端从 cookie 读取；前端不再接触 token。
- **方案 B（轻量）**：维持 localStorage，但缩短有效期 + 引入 refresh token + 全站 CSP 防 XSS。

**验收标准**：
- [ ] XSS 无法直接读取长期凭证
- [ ] 登录/登出/刷新链路回归通过
- [ ] 前端 API 契约不变

**影响面**：frontend、backend（auth）

---

### 任务 2.3　源码编码治理（P1-4）

**方案**：
1. 修复 `services/restClient.ts` 等被 GBK 字节污染的中文注释，统一转为 UTF-8。
2. 全仓扫描：`grep -rlP '[\x80-\xff]'` + 人工核对乱码文件。
3. 新增 `.editorconfig`（`charset = utf-8`）。
4. CI/`verify.sh` 加 UTF-8 合法性校验（非法字节即失败）。

**验收标准**：
- [ ] 全仓中文注释正常显示，无 mojibake
- [ ] CI 含编码校验闸门
- [ ] `.editorconfig` 就位

**影响面**：frontend、CI、tooling

## 阶段三：中期（工程债 / 可维护性）

### 任务 3.1　仓库卫生治理（P2-1）

**方案**：
1. 补全 `.gitignore`：`node_modules/`、`.opencode/`、`__MACOSX/`、`*.rar`、AI 工具产物目录。
2. `gorm/` vendored 源码改走 `go.mod` 依赖（若非有意 fork）。
3. 清理 `新建文件夹/`（HAR/截图）—— 移到独立的"逆向资料"归档，不进主代码仓。
4. 评估 `.qodo`/`.sisyphus`/`thoughts/` 是否需入库。

**验收标准**：
- [ ] 仓库不含已安装依赖与平台垃圾文件
- [ ] 全新 clone 后 `go build` / `npm i` 能复现依赖
- [ ] 打包体积大幅下降

**影响面**：全仓、tooling

---

### 任务 3.2　收敛 v1/v2 架构边界（P2-2）

**方案**：
1. 明确文档化：上机/下机的**唯一事实源**（主系统 `services/treatment_service.go` 还是 `smart_schedule/service/treatment_service.go`），另一方仅作只读/同步消费者。
2. 梳理两份 `treatment_service.go` 的"同步钩子"，消除双写竞态。
3. 中期评估是否将 v2 repo 范式与 v1 service 范式对齐（不强求统一，但需边界清晰）。

**验收标准**：
- [ ] 产出《v1/v2 边界与事实源说明》
- [ ] 上机/下机不存在双写同一状态的竞态
- [ ] 同步钩子有测试覆盖

**影响面**：backend、docs

---

### 任务 3.3　前端引入最小化测试（P2-3）

**方案**：
1. 接入 Vitest + @testing-library/react。
2. 优先覆盖高风险临床录入流程：透前评估、透析医嘱、双人核对、上机/下机。
3. 为 `services/*Api.ts` 请求层加单测（mock axios）。
4. `npm run build` 闸门后追加 `npm run test`。

**验收标准**：
- [ ] 核心录入流程有冒烟级测试
- [ ] 请求层关键封装有单测
- [ ] CI 跑前端测试

**影响面**：frontend、CI

## 执行约定

- 提交遵循 Conventional Commits（`feat:`/`fix:`/`chore:`/`refactor:`），PR 标注影响面（backend/frontend/deploy/docs）、验证方式、新增环境变量。
- 每个任务独立 PR、独立可回滚；改动老库相关逻辑前先核对 `新数据库表结构.md` 与 `老血透数据库表结构-合并版.md`，**禁止凭字段名猜测映射**。
- 提交前必跑：后端 `./scripts/verify.sh`（gofmt+vet+build）+ `./scripts/smoke_test.sh`；前端 `npm run lint` + `npm run build`。
