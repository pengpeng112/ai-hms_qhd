# AI-HMS 透析中心管理系统 — 系统评估报告

> 评估日期：2026-06-10
> 评估范围：`ai-hms-backend`（Go 后端）、`ai-hms-frontend`（React 前端）、相关文档与部署配置
> 评估方式：源码静态走查 + 文档交叉核对 + 关键路径采样
> 结论摘要：**功能完整度高、工程规范在水准之上的真实临床系统**；最大风险不在代码质量，而在"直连老生产库 + 运行时 DDL"这一架构红线被自身破坏，以及围绕老库的数据安全保障是否到位。

## 一、系统概述

### 1.1 定位

一套**血液透析中心管理系统（AI-HMS）**，采用"边迁移边替换"策略：应用直接连接并**并行运行**于一个**老血透生产 PostgreSQL 库**之上，逐页/逐模块用新系统替换老系统，同时保持前端 API 契约不变、所有适配在后端完成。

### 1.2 技术栈与规模

| 子工程 | 技术栈 | 规模 |
|---|---|---|
| `ai-hms-backend` | Go 1.24 + Gin 1.11 + GORM 1.31 + PostgreSQL(pgx) + JWT + snowflake；HDIS/LIS 外部对接（含 chromedp 无头浏览器抓 token） | ~39,400 行 Go |
| `ai-hms-frontend` | React 19 + TypeScript 5.9 + Vite 7 + Ant Design 6 + Tailwind 4 + Redux + i18next + recharts + xlsx | ~41,300 行 TS/TSX |
| `新建文件夹/old_system` | 老系统 HAR 抓包 + 截图（用于逆向 API 契约/字段语义） | — |

### 1.3 量化指标

| GORM 模型文件 | 15 |

```
前端 Nginx(:3000)  --反代 /api/-->  后端 Gin(:8080)  --->  外部 PostgreSQL(10.10.8.83 老血透生产库)
```

## 二、功能模块全景

### 2.1 临床主流程（v1，`/api/v1`）

与老系统截图（`old_system/2.x 透析执行—*`）一一对应，覆盖透析执行全链路：

| 模块 | 说明 |
|---|---|
| 患者管理 | 基础信息 / 核心信息聚合 / 住院信息 / 临床病史 / 关键指标 |
| 透析执行闭环 | 透前评估 → 当日处方 → 双人核对 → 透析医嘱 → 透中监测 → 透后评估 → 健康宣教 → 透析小结 |
| 检验/检查报告 | 含 HDIS / LIS 外部系统同步 |
| 资源管理 | 病区 / 床位 / 设备 / 库存 / 耗材 |
| 运营视图 | 监测大屏、统计、月度小结、Dashboard、临床任务 |
| 系统管理（admin 组） | 字典、用户、角色、权限、HDIS 集成配置、系统日志 |

### 2.2 智能排班（v2，`/api/v2`，`internal/smart_schedule/`）

一个**相对独立、自带 model/repo/service/engine 的子系统**，由独立参考后端（v1.4）移植而来：

- 14 张排班表、两轮分配引擎（`sched/engine.go`）、奇偶周计算（`sched/week.go`）、新病人初始化（`sched/newpatient.go`）
- 13 个业务服务：排班生成、周看板、差异、质量评分、健康检查、运营操作（确认/取消/缺席/移床）、CRRT、补透、停机/假日扰动、生命周期（入科/出组/院感）、模板、上机/下机+院感闸门、管理后台

**评价**：功能覆盖完整，临床闭环与排班引擎均已实质落地，并非骨架 demo。

## 三、架构评估

### 3.1 后端分层

```
api/v1/*_handler.go   Gin handler，参数解析 + 响应封装
services/*.go         业务逻辑、GORM 查询、事务
models/*.go           GORM 实体（表名映射到老库历史表）
database/             连接（database.go）；migrate.go 为禁用占位
middleware/           CORS、JWT 鉴权、请求日志
integrations/hdis/    HDIS 外部系统 HTTP/GraphQL 对接
utils/                JWT、密码哈希、snowflake ID
pkg/response/         统一响应体 {success, data/error, timestamp}
```

分层清晰，路由集中在 `cmd/server/main.go` 通过 `RegisterXxxRoutes` 注册。三层职责分离规范，**工程化程度高于一般行业系统**。

### 3.2 鉴权与权限模型

- JWT（`golang-jwt/jwt/v5`），登录公开、其余路由挂 `AuthMiddleware`
- claims 含 `UserID / Username / Roles / TenantID`，缺 tenant_id 直接 403
- admin 路由组用 `RequireRoles` 做角色门禁
- 前端有 `PermissionGuard` 组件 + 后端 permission 模块（细粒度权限码已存在）

### 3.3 数据访问

- GORM `SingularTable + NoLowerCase` 命名策略，对齐老库历史表名
- `PrepareStmt: true`、连接池、慢 SQL 阈值、`IgnoreRecordNotFoundError`
- 启动时强制 `Ping` + `SELECT 1` + `SET client_encoding=UTF8` 自检
- `AutoMigrate` 已被永久禁用（`migrate.go` 返回固定错误）

### 3.4 值得肯定之处

1. **测试与闸门**：后端 16 个测试文件，`verify.sh`（gofmt+vet+build）+ `smoke_test.sh` 提交前闸门
2. **老库守则成文**：禁 AutoMigrate、字段映射查文档不靠猜，"待确认字段"沉淀为清单
3. **生产化细节到位**：统一错误响应、连接池、健康检查含 DB ping、HDIS token 管理、参数化查询开关、日志按级别分流（app.log / error.log）
4. **文档扎实**：CLAUDE.md（含 GitNexus 索引）、排班交接文档、数据库设计、ER 图、字典对照表

## 四、问题与风险清单

> 严重程度：🔴 P0 阻断/高危 ｜ 🟠 P1 需尽快处理 ｜ 🟡 P2 中期改进

### 🔴 P0-1：运行时 DDL 破坏了自定的"老生产库红线"

**现象**：全局硬规则三处强调"运行时严禁 DDL / 建表 / 改表"，`AutoMigrate` 已永久禁用。但智能排班启动注册时：

`internal/smart_schedule/api/api.go:41` → `repo.EnsureIndexes(s.DB)` → `internal/smart_schedule/repo/repo.go:116`：

```sql
CREATE UNIQUE INDEX IF NOT EXISTS uq_ps_patient_slot ...
CREATE UNIQUE INDEX IF NOT EXISTS uq_ps_machine_slot ...
```

**风险**：
- `CREATE UNIQUE INDEX` 本身即 DDL，直接对老生产库执行，与既定红线矛盾
- 唯一索引可能因老库**存量脏数据/重复行**建失败，导致服务启动失败
- 注释自称"兼容禁止 AutoMigrate 的约束"，但这只是绕过了 GORM AutoMigrate，并未绕过"DDL"本质

**定位**：`internal/smart_schedule/api/api.go:41`、`internal/smart_schedule/repo/repo.go:113-124`

### 🔴 P0-2：直连老生产库并行写入（固有架构风险）

应用直接读写老系统生产库、与老系统同时运行。这是项目最大的固有风险：

- 并发写冲突、双系统同时改同一行
- 无迁移机制 → schema 漂移靠人工对齐
- 回滚仅靠"一次只迁一个模块"的纪律

架构上无法消除，但**必须配套**：写操作审计、关键表唯一约束/乐观锁、灰度开关。需确认这些保障是否齐备。

### 🟠 P1-1：角色校验硬编码中文角色名

`cmd/server/main.go:171`：

```go
middleware.RequireRoles("ADMIN", "管理员", "安全管理员", "运维管理员")
```

**风险**：角色名改动即权限失效；中英文混排易漏配；权限逻辑散落在路由注册处而非集中策略。已有细粒度 permission 模块却未用于此处门禁。

### 🟠 P1-2：JWT 存储于 localStorage

前端 `services/restAuth.ts` + `utils/token` 将 accessToken 存入 localStorage，易受 XSS 窃取。Antd 富交互 + 大量数据录入页面，XSS 面不小。

### 🟠 P1-3：演示账号 / 预览旁路登录

- 历史演示账号 `admin / admin123`（CLAUDE.md 承认"可能仍可用"）
- 前端 `restAuth.ts:98` 存在 `local-preview-token` 旁路登录路径

需确认生产构建已彻底关闭这两条旁路。

### 🟠 P1-4：源码编码损坏（UTF-8/GBK 混入）

`services/restClient.ts:10` 等处中文注释为 GBK 字节混入 UTF-8 文件：

```
// API 閰嶇疆        ← 实为 "// API 配置"
// 鏍囧噯鍝嶅簲鏍煎紡  ← 实为 "// 标准响应格式"
```

同文件首部注释却正常，说明有人在非 UTF-8 编辑器改过文件。当前波及约 2 个文件，但若不治理会扩散。

### 🟡 P2-1：仓库卫生差

打包内包含大量不应入库的内容：

- `ai-hms-frontend/node_modules`、`.opencode/node_modules`（已安装依赖）
- 整份 `gorm/` 源码树（应走 go.mod）
- `__MACOSX/`、`新建文件夹/`、`.qodo/`、`.sisyphus/`、`thoughts/`（AI 工具产物）

导致 138MB 压缩包绝大多数是噪音。

### 🟡 P2-2：v1 / v2 两套架构范式并存

- `internal/services/*`（主系统，GORM 直查）
- `internal/smart_schedule/{repo,service,sched}`（自带 repo 层 + 引擎）

两种范式 + 各有一份 `treatment_service.go` 通过"同步钩子"耦合。上机/下机的唯一事实源不清晰，长期维护需在两套心智模型间切换。

### 🟡 P2-3：前端零自动化测试

前端 0 个测试文件，4 万行 TS、临床数据录入场景全靠手动验证，回归风险高。

### 🟡 P2-4：迁移文档分散

`legacy-migration-pending-confirmation.md` / `confirmed-completed.md` / `uncertain-field-checklist.md` / `LEGACY_TABLE_FIELD_MAPPING.md` 多份并存，缺单一可追踪台账。
