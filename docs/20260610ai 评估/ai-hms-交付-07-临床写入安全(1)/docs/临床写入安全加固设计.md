# 临床写入安全加固设计（乐观锁 / 幂等 / 审计）

> 制定日期：2026-06-10
> 上游依据：《老库写操作保障核查.md》——核查结论为"无乐观锁、无悲观锁、幂等仅排班一处、无 DB 业务审计"。
> 适用前提：应用与老系统**并行直写同一 PostgreSQL 生产库**；遵守"运行时严禁 DDL、表结构变更走外部 SQL + DBA"。
> 设计原则：**前端 API 契约（路径/字段/响应体）保持不变**；一次只动一个表、向后兼容、可回滚、可灰度。

`新数据库表结构.md` 与 `老血透数据库表结构-合并版.md` 确认列真实存在与类型，禁止凭模型字段名直接写库。

## 1. 目标与风险对齐

| 风险（来自核查） | 本设计对策 | 严重度 |
|---|---|---|
| Lost update（并发覆盖写） | 乐观锁（版本令牌条件更新，冲突返回 409） | 🔴 高 |
| 重复业务记录（重试/双击） | 幂等键（自然键唯一约束 / 幂等表） | 🟠 中 |
| 不可追溯（无业务审计） | 结构化写审计（应用层审计表 + 评估 DB 触发器） | 🟠 中 |

三类对策相互独立，可分别灰度。**优先级：乐观锁 > 幂等 > 审计**（按"防止错误数据"紧迫度排序）。

## 2. 乐观锁设计

### 2.1 令牌选择（按表能力分两档）

- **A 档（首选，最稳）——整型版本列**：`UPDATE … SET …, "Version"="Version"+1 WHERE "Id"=? AND "Version"=?`。
  命中 0 行即冲突。整型自增无精度问题。
  现成可用：`Schedule_ScheduleTemplate`。其余热点表若要用此档，需 DBA 经外部 SQL 加
  `"Version" int NOT NULL DEFAULT 0` 列（见 §2.4），**不在代码路径建列**。

- **B 档（过渡，零改表）——时间戳列**：`UPDATE … SET … WHERE "Id"=? AND "LastModifyTime"=?`。
  现成可用：`Treatment_Treatment`、`Schedule_PatientShift`。
  ⚠️ 注意：
  1. 时间戳相等比较存在**精度风险**（Postgres `timestamp` 微秒 vs Go `time.Time` 纳秒，读出与回填须保证同一精度/时区）；
  2. `LastModifyTime` 为 `autoUpdateTime`，更新时 ORM 会自动改写，**条件用的是更新前读到的旧值**，逻辑成立；
  3. `RowsAffected==0` 同时覆盖"已被改"与"已被删"，统一按 409 处理（语义：请重新加载）。

> 结论：先用 B 档让三张热点表立即获得乐观锁；对最高频/最关键的表（建议 `Treatment_Treatment`）
> 推进 A 档整型版本列，彻底消除时间戳精度隐患。

### 2.2 与前端契约兼容：用 HTTP ETag / If-Match（不改 JSON 契约）

JSON 响应体**不新增字段**，版本令牌走 HTTP 头：

1. **读接口（GET）**：响应头返回弱 ETag，由版本令牌编码而来，例如
   `ETag: W/"v7"`（整型版本）或 `ETag: W/"t1718000000000000000"`（时间戳 unix 纳秒）。
2. **写接口（PUT/POST 修改）**：客户端回传 `If-Match: W/"v7"`。
   - 命中 → 条件更新；冲突 → **412 Precondition Failed** 或 **409 Conflict**（二选一，统一约定，建议 409 便于前端区分"业务冲突"）。
   - **缺失 `If-Match`（过渡期）**：可配置为"放行（兼容旧前端，回退为最后写入者胜）"或"拒绝（严格）"。
     上线初期设为**放行**，前端 `restClient.ts` 统一改造为透传 ETag 后，再切**严格**。

> 该方案使**前端只需在 axios 单例 `restClient.ts` 一处**做"GET 存 ETag → 写回 If-Match"的通用拦截，
> 业务页面与字段结构零改动，符合"契约不变、改动收敛后端 + 单点前端"。

### 2.3 服务层流程（参考）

```
读：service.Get(id) -> 返回实体 + version 令牌 -> handler 写入 ETag 头
写：handler 解析 If-Match -> service.Update(id, expectedVersion, patch)
      -> clinicalsafety.UpdateWithVersion / UpdateWithTimestamp
      -> RowsAffected==0 -> VersionConflictError -> handler 映射 409
```

参考实现见 `internal/clinicalsafety/`（本次随设计交付，**独立新包，未改动任何在迁移中的文件**）。

### 2.4 外部 SQL（A 档加列，DBA 执行；示例，需 schema 确认后定稿）

```sql
-- 仅示例：为高频表增加整型版本列（确认列不存在后执行）
ALTER TABLE "Treatment_Treatment" ADD COLUMN IF NOT EXISTS "Version" integer NOT NULL DEFAULT 0;
-- 回滚： ALTER TABLE "Treatment_Treatment" DROP COLUMN IF EXISTS "Version";
```

## 3. 幂等设计

针对**创建型**临床写入（上机/下机、医嘱下达、补透、临时透析）防重试/双击重复落库。

### 3.1 两种手段（优先自然键，其次幂等表）

- **手段一：自然键唯一约束（首选，零新表）**
  已有先例：排班 `uq_ps_patient_slot`/`uq_ps_machine_slot`（见 `scripts/schedule_unique_indexes.sql`）。
  对存在天然唯一语义的创建操作，补唯一索引 + `OnConflict{DoNothing}`/捕获唯一冲突即可幂等。
  新增唯一索引一律走**外部 SQL + DBA + 重复数据探测**（同 1.1 套路）。

- **手段二：幂等键表（无自然键时）**
  客户端在创建请求头带 `X-Idempotency-Key`（或 body `clientRequestId`）。新增**独立**表：

  ```sql
  -- 新表，不改动任何既有表；DBA 执行
  CREATE TABLE IF NOT EXISTS "App_IdempotencyKey" (
    "TenantId"   bigint      NOT NULL,
    "Scope"      varchar(64) NOT NULL,   -- 操作类型，如 'treatment.start'
    "Key"        varchar(128) NOT NULL,  -- 客户端幂等键
    "ResultRef"  bigint,                 -- 已生成业务记录主键，便于重放返回
    "CreatedAt"  timestamptz NOT NULL DEFAULT now(),
    PRIMARY KEY ("TenantId","Scope","Key")
  );
  ```

  流程：写操作开始前 `INSERT ... ON CONFLICT DO NOTHING`；插入成功→执行业务并回填 `ResultRef`；
  插入冲突→说明重复请求，直接返回首次结果（幂等重放）。定期清理过期键（TTL，由运维 job 或 DBA）。

### 3.2 契约兼容

`X-Idempotency-Key` 为**可选**请求头：带则启用幂等，不带则维持现状（不破坏旧前端）。
前端同样可在 `restClient.ts` 单点为"创建型"请求生成并附带该头。

## 4. 审计设计

现状：仅文件日志（`log_service` 读 app.log/error.log），无结构化业务审计，临床改动不可逐条还原。

### 4.1 应用层审计（覆盖"本系统"写入，现在即可做）

新增**独立**审计表（DBA 经外部 SQL 建，不动既有表）：

```sql
CREATE TABLE IF NOT EXISTS "Audit_ClinicalWrite" (
  "Id"         bigserial PRIMARY KEY,
  "TenantId"   bigint NOT NULL,
  "OperatorId" bigint,
  "Action"     varchar(64) NOT NULL,   -- 'treatment.start' / 'order.update' ...
  "TableName"  varchar(64) NOT NULL,
  "RecordId"   bigint,
  "Before"     jsonb,                  -- 变更前快照（可空）
  "After"      jsonb,                  -- 变更后快照
  "RequestId"  varchar(128),           -- 关联请求/幂等键
  "CreatedAt"  timestamptz NOT NULL DEFAULT now()
);
```

实现方式：在关键临床 service 写路径，于同一事务内追加一条审计写入（操作人取自 JWT claims，
前后值取自更新前后的实体快照）。封装为可复用的 `AuditWriter`，避免逐方法重复。

### 4.2 数据库触发器（覆盖"老系统并行写入"，需评估）

由于老系统也直写同库，**仅应用层审计无法捕获老系统的改动**。若需全覆盖审计，
唯一可靠路径是 DBA 在关键表上加触发器写 `Audit_ClinicalWrite`。
代价：触发器属库内对象、由 DBA 维护，需评估对老系统写入性能与既有触发器的影响。
**建议**：先做应用层审计（本系统改动可追溯），并行评估触发器方案以补齐老系统侧覆盖。

## 5. 落地顺序（每步独立 PR、可回滚、向后兼容）

1. **参考实现合入**（本次已附 `internal/clinicalsafety/`，纯新增、带单测）。
2. **试点一张表 = `Treatment_Treatment` 上/下机**：
   - service 读路径返回版本令牌；handler 输出 ETag。
   - 写路径接 `UpdateWithTimestamp`（B 档），`If-Match` 缺失时放行（兼容）。
   - handler 将 `VersionConflictError` 映射 409。
3. **前端 `restClient.ts` 单点透传 ETag/If-Match**（GET 存、写回带），灰度后将试点表切"严格"。
4. **幂等试点 = 上机/下机**：加 `X-Idempotency-Key` 支持（自然键优先）。
5. **审计试点 = 上机/下机 + 医嘱更新**：事务内写 `Audit_ClinicalWrite`。
6. 验证稳定后，按 `orders`/`prescriptions`/`Schedule_PatientShift` 逐表复制，并为 `Treatment_Treatment` 推进 A 档整型版本列。

## 7. 风险与权衡

| 决策点 | 取舍 |
|---|---|
| 时间戳 vs 整型版本 | 时间戳零改表但有精度风险；整型版本最稳但需 DBA 加列。先时间戳起步，热点表升整型。 |
| 409 vs 412 | 412 更符合 HTTP 语义；409 更易被前端按"业务冲突"统一处理。建议 409 并文档化。 |
| 应用层审计 vs 触发器 | 应用层即时可做但漏老系统写入；触发器全覆盖但 DBA 成本与性能需评估。两者互补。 |
| If-Match 缺失放行 vs 拒绝 | 放行保证旧前端不被破坏（过渡），前端单点改造完成后切拒绝。 |
