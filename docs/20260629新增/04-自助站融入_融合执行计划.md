# 自助站融入融合执行计划

## 1. 通俗功能说明

这个项目包含两部分：一个较大的自助站桌面程序 `app_self`，以及 AI-HMS 前端可见性融合补丁。自助站做人脸签到、体重、血压、体温等自助采集；AI-HMS 侧负责让护士和医生看到“已签到、已自测、待体征”。

简单说：患者在自助机上签到和测量后，护士和医生在 AI-HMS 里能一眼看到人到了、体征有了、医生是否可以开方。

## 2. 当前系统状态

- 自助站数据定位结果显示，体重/血压/体温写入 `Treatment_BeforeSigns`。
- 最近 7 天 `Treatment_BeforeSigns` 有 301 行，说明数据在活跃写入。
- 2026-06-30 只读核实：`Treatment_BeforeSigns` 当前 23925 行 / 23925 个治疗，没有同治疗多行。
- 2026-06-30 只读核实：`Treatment_BeforeSigns` 存在唯一约束 `treatment_beforesigns_un UNIQUE ("TreatmentId")`，因此同一次治疗不能直接向该表追加多行。
- 当前 AI-HMS 透前评估已读取 `beforeSigns`，但缺少“这是已带入体征，请核对”的提示。
- 当前工作台 `Cockpit.tsx` 尚未显示“已自测/已签到/待体征”。
- `app_self` 下有源码、pyc、日志、build、dist、exe、dll 等，不应整体合入当前系统。

## 2.1 用户已确认决策

- 自助站继续实时获取设备体征数据，但后续应由 AI-HMS 获取后存储到老库。
- 重复上传同一次治疗体征时，业务要求是“追加”，不是拒绝。
- 允许在 `Treatment_BeforeSigns.Note` 写入来源标识，标识值使用 `newsystem`。
- 因老表 `Treatment_BeforeSigns` 对 `TreatmentId` 有唯一约束，直接追加到该表会失败；融合设计必须用“追加明细表 + 旧表最新值镜像”的方式满足业务要求。
- 用户提供 `Treatment_Treatment` 状态码表后确认：患者签到应写治疗记录本身，`Status=10` 表示“已签到”，并可同步写 `SignInTime`。
- 2026-06-30 只读核实与代码复核补充：当前生产库没有 `Status='10'` 样例；当前后端 `legacyStatusFromApp` 对待签到/默认态写 `Status='0'`，且治疗创建时会写 `SignInTime`。因此为尽量保持现有数据一致，患者“已签到”的 UI 判断应优先使用 `SignInTime IS NOT NULL`，`Status='10'` 仅作为兼容旧码表的辅助判断。
- 2026-06-30 代码复核补充：当前 `TreatmentRealtimeResponse` 没有 `json:"signInTime"` 字段，Cockpit 当前拿不到小写 `signInTime`；一期前端若要按 `SignInTime` 判断已签到，需要后端列表响应补出该字段。

## 3. 融合原则

- 不合入 `app_self` 大程序、打包产物、日志、二进制文件。
- 本期只做 AI-HMS 侧接口和可见性融合。
- 既有自助站历史数据可继续由 AI-HMS 只读消费；二期新接口按用户确认改为“自助站调用 AI-HMS API，AI-HMS 负责写入老库”。
- AI-HMS 写入接口必须单独做认证和开关控制，不能默认开放匿名写入。

## 4. 建议分两期执行

### 4.1 一期：消费已落库数据，做 UI 可见性

改动范围：纯前端。

文件：

- `ai-hms-frontend/src/pages/Cockpit.tsx`
- `ai-hms-frontend/src/pages/dialysis-processing/execution/PreAssessment.tsx`

功能：

- 护士床位墙显示：
  - `已自测`：`beforeSigns.weight > 0` 或 `beforeSigns.sbp > 0`
  - `已签到`：优先 `signInTime` 非空；兼容判断 `legacyStatus >= 10`
- 透前评估顶部提示：有自测体征时显示“本次透前体征已带入，请核对后保存”。
- 医生分诊墙新增“待体征”：
  - 体重未录入且处方未签发时显示。
  - 体重已录入后转入待开方/待签/已签。
  - 已签发处方不被降级为待体征。

### 4.2 二期：只做接口，供自助站独立部署后调用

用户已确认需要做接口，并将设备实时体征数据存储到老库。由于 `Treatment_BeforeSigns` 对 `TreatmentId` 唯一，不能直接追加多行，建议二期采用下面的双写策略：

- 新增独立明细表追加保存每次自助站测量，保留完整设备历史。
- 同步最新一条到 `Treatment_BeforeSigns`，供当前 AI-HMS 既有透前评估、工作台、处方前置逻辑继续读取。
- `Treatment_BeforeSigns.Note` 写 `newsystem`，用于标记来源。
- 新表 DDL 只能写入部署 SQL，由 DBA/部署阶段执行；应用运行时不得建表。

建议 API 设计：

| 方法 | 路径 | 用途 |
|---|---|---|
| `POST` | `/api/v1/kiosk/check-in` | 自助站签到 |
| `POST` | `/api/v1/kiosk/pre-signs` | 自助站上传透前体重/血压/体温 |
| `GET` | `/api/v1/kiosk/patients/lookup` | 通过人脸/透析号/身份证查患者和当天治疗 |
| `GET` | `/api/v1/kiosk/health` | 自助站接口健康检查 |

建议新增独立明细表：`kiosk_pre_sign_measurement`。

建议字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | varchar/bigserial | 主键 |
| `tenant_id` | bigint | 租户，固定 `LegacyTenantID` |
| `treatment_id` | bigint | 治疗 ID |
| `patient_id` | bigint | 患者 ID |
| `measured_at` | timestamptz | 设备测量时间 |
| `weight` | numeric | 体重 |
| `sbp` | numeric | 收缩压 |
| `dbp` | numeric | 舒张压 |
| `body_temp` | numeric | 体温 |
| `heart_rate` | numeric | 心率 |
| `respiration` | numeric | 呼吸 |
| `device_id` | varchar | 设备编号，可空 |
| `source` | varchar | 固定 `newsystem` |
| `raw_payload` | jsonb/text | 原始上报，可选 |
| `created_at` | timestamptz | 入库时间 |

安全要求：

- 新增 `KIOSK_API_ENABLED=false`，默认关闭。
- 新增 `KIOSK_API_TOKEN` 或 HMAC 签名，不允许匿名写入。
- 写入必须固定 `TenantId=LegacyTenantID`。
- 明细表按业务要求追加保存，不拒绝重复测量。
- 为防止网络重试造成完全重复，可支持可选 `client_event_id` 幂等键；没有幂等键时按追加处理。
- `Treatment_BeforeSigns` 只保存同治疗最新镜像，因为该表 `TreatmentId` 唯一。
- `Treatment_BeforeSigns.Note` 写入 `newsystem` 标记来源。

## 5. 一期具体执行步骤

### 5.1 修改 `Cockpit.tsx`

1. `DoctorState` 增加 `needVitals`。
2. `DOCTOR_META` 增加 `needVitals` 样式，并调整排序。
3. `BedCard` 增加 `kioskCheckedIn`、`kioskSelfMeasured`。
   - **注意命名冲突**：当前 `Cockpit.tsx:126` 已有 `const [checkedIn, setCheckedIn]`，它追踪的是护士排班签到（smart schedule duty system），与患者签到是不同概念。因此 `BedCard` 新增的字段必须用 `kioskCheckedIn`/`kioskSelfMeasured`，避免与现有 `checkedIn` 混淆。
4. 增加纯函数：
   - `parseLegacyCheckedIn(legacy string) bool`
   - `hasSelfSigns(before) bool`
5. 在护士床卡数据映射时派生 `kioskCheckedIn` 和 `kioskSelfMeasured`。
   - `kioskCheckedIn` 推荐逻辑：`!!t.signInTime || parseLegacyCheckedIn(t.legacyStatus || '')`。
   - 当前 `TreatmentRealtimeResponse` 尚未暴露 `signInTime`，需要先在后端响应结构、查询映射、前端类型中补出 `SignInTime`/`signInTime`，避免只靠 `Status='10'` 误判。
6. 在医生卡态判断中优先判断 `needVitals`，但不要覆盖已签处方。
7. 在护士床卡 chip 旁显示小角标：优先 `已自测`，其次 `已签到`。

### 5.2 修改 `PreAssessment.tsx`

1. 在 return 前计算 `selfMeasured`。
2. `treatmentLoading` 为 false 且 `selfMeasured` 为 true 时，显示提示条。
3. 不改变现有表单预填、保存逻辑。

## 6. 二期接口具体执行步骤（已按用户确认调整）

后端建议新增：

- `internal/api/v1/kiosk_handler.go`
- `internal/services/kiosk_service.go`
- `config.KioskConfig`

写入策略建议：

- `/kiosk/pre-signs` 查当天治疗，找到 `Treatment_Treatment.Id`。
- 每次上报先向 `kiosk_pre_sign_measurement` 追加一行，保留完整测量历史。
- 然后同步最新值到 `Treatment_BeforeSigns`：
  - 若该治疗无 `Treatment_BeforeSigns` 行，则插入一行。
  - 若该治疗已有行，则更新该行的 Weight/SBP/DBP/BodyTemp/HeartRate/Respiration/OperateTime/LastModifyTime/Note。
  - `OperateTime` 应使用设备上传的 `measured_at`，而非 `now()`，保持时间戳含义与现有数据一致（现有 301 行均填设备测量时间）。
  - **禁止改动** `Id`、`CreateTime`、`CreatorId`，保护首次录入审计链。
  - `Note` 写 `newsystem`，如需保留原备注，可使用追加格式，例如 `原备注; source=newsystem`。
- `/kiosk/check-in` 不写 `Schedule_CheckIn`，因为当前定位显示它更像护士接班。
- `/kiosk/check-in` 更新 `Treatment_Treatment`：
  - 为与当前生产库和现有 Go 侧状态映射保持一致，签到接口**优先只补写 `SignInTime`** 和 `LastModifyTime`。
  - 当前 `Status` 为空或 `0` 时，默认仍保留 `Status='0'`，不要强制改为 `10`，除非后续确认旧系统/HDIS 依赖 `10` 才能识别已签到。
  - 若当前 `Status='10'`，视为兼容旧码表的已签到状态，不回退。
  - 若当前 `Status` 已为 `20/30/40/50/60` 或生产库扩展码（如 `70/90/100/110/120/130/150`），不得把状态倒退；只在 `SignInTime` 为空时补写签到时间。
  - 不改 `ReceptionTime`、`StartTime`、`EndTime`，避免干扰接诊、上机、下机流程。
  - `Status='50'` 中断、`Status='60'` 已结束时，接口应返回已完成/不可重复签到提示，不再改状态。

## 8. 尚需确认的问题

- 是否需要 `client_event_id` 防止自助站网络重试造成重复明细？如果不提供，则每次请求都会追加。
- `Treatment_BeforeSigns.Note` 已确认可写 `newsystem`，但是否需要保留原 `Note` 文本并追加来源标识？
- 是否仍要求接口把 `Status='0'` 强制改为 `Status='10'`？当前生产库和 Go 侧默认态更接近 `Status='0' + SignInTime`，直接改 `10` 可能引入与现有状态映射不一致的问题。

推荐默认：

- 建议自助站提供 `client_event_id`，避免设备/网络重试导致完全重复明细；若设备暂时无法提供，则按业务要求每次请求都追加。
- 建议更新旧表镜像时保留原 `Note` 并追加 `source=newsystem`，不要无条件覆盖人工备注。
- 推荐签到写法：优先补 `SignInTime`，不强制把 `Status='0'` 改成 `10`；前端以 `SignInTime` 判断“已签到”。
