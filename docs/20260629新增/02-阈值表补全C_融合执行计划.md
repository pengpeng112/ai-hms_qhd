# 阈值表补全 C 融合执行计划

## 1. 通俗功能说明

当前实时监控已能报警，但固定阈值还只覆盖平均动脉压、心率、超滤率、透析液钠等少数指标。这个项目补齐 5 个常见监测指标：血氧、呼吸、跨膜压 TMP、动脉压 AP、电导率。

简单说：让床卡不仅能看血压心率，也能对血氧低、呼吸异常、管路负压过高、TMP 过高、透析液电导率异常发出提示。

## 2. 当前系统状态

- 当前 `internal/config/monitoring_thresholds.json` 尚未包含 `spo2`、`respiration`、`tmp`、`ap`、`conductivity`。
- 当前 `evalAlarms` 尚未评估这 5 个指标。
- 当前阈值 admin 页是通用 fixed 表，恢复默认后理论上可自动显示新指标，无需改前端。
- 当前线上 DB 若已有阈值表数据，新增 JSON 种子不会自动插入 DB，需通过 admin “恢复默认”或 DBA 种子脚本处理。

## 3. 融合目标

- 在默认 JSON 阈值中新增 5 个指标。
- 在实时监控报警引擎中评估 5 个指标。
- 补充单元测试，防止阈值口径回归。
- 不改前端页面、不改 API 契约。

## 4. 具体改动文件

- `ai-hms-backend/internal/config/monitoring_thresholds.json`
- `ai-hms-backend/internal/services/monitoring_service.go`
- `ai-hms-backend/internal/config/monitoring_threshold_test.go`

## 5. 指标定义

| 指标 | key | 判定口径 |
|---|---|---|
| 血氧饱和度 | `spo2` | `<90` 危险，`90-94` 警戒，`>=95` 正常 |
| 呼吸 | `respiration` | `<8` 危险，`8-11` 警戒，`12-20` 正常，`21-25` 警戒，`>25` 危险 |
| 跨膜压 TMP | `tmp` | `<=250` 正常，`250-350` 警戒，`>350` 危险 |
| 动脉压 AP | `ap` | 按绝对值判断，`<=200` 正常，`200-250` 警戒，`>250` 危险 |
| 电导率 | `conductivity` | `<13.0` 危险，`13.0-13.5` 警戒，`13.5-14.5` 正常，`14.5-15.0` 警戒，`>15.0` 危险 |

## 6. 执行步骤

### 6.1 更新阈值 JSON

在 `fixed` 中新增：

```json
"spo2": {"label":"血氧饱和度","enabled":true,"unit":"%","dangerLow":90,"warnLow":95},
"respiration": {"label":"呼吸","enabled":true,"unit":"次/分","dangerLow":8,"warnLow":12,"warnHigh":20,"dangerHigh":25},
"tmp": {"label":"跨膜压TMP","enabled":true,"unit":"mmHg","warnHigh":250,"dangerHigh":350},
"ap": {"label":"动脉压AP","enabled":true,"unit":"mmHg","warnHigh":200,"dangerHigh":250},
"conductivity": {"label":"电导率","enabled":true,"unit":"mS/cm","dangerLow":13.0,"warnLow":13.5,"warnHigh":14.5,"dangerHigh":15.0}
```

注意保持 JSON 合法逗号。

### 6.2 更新报警引擎

在 `monitoring_service.go`：

1. 增加 `math` import。
2. 在 `evalAlarms` 中加入：
   - `d.SpO2 > 0` 时评估 `spo2`。
   - `d.Respiration > 0` 时评估 `respiration`。
   - `d.TMP > 0` 时评估 `tmp`。
   - `d.ArterialPressure != 0` 时取 `math.Abs(d.ArterialPressure)` 后评估 `ap`。
   - `d.Conductivity > 0` 时在现有 `dialysateNa` 后追加 `conductivity` 原值评估。
3. 不要改变已有 MAP、heartRate、VP、UFR 逻辑。

### 6.3 更新测试

在 `monitoring_threshold_test.go` 增加：

- `TestEvalFixed_SpO2_LowOnly`
- `TestEvalFixed_Respiration`
- `TestEvalFixed_TMP_HighOnly`
- `TestEvalFixed_AP_HighOnly`
- `TestEvalFixed_Conductivity`

## 7. 数据库与运营动作

- 不需要新增 DDL。
- 对已部署的 DB 阈值表，新增 JSON 不会自动进入 DB。
- 上线后需要管理员在阈值页面点击"恢复默认"，或由 DBA 按三张阈值表结构补种这 5 个指标。
- **重要**：当前 `ResetThresholdAdmin` 的实现是删整表再按 JSON 重建（`writeAllThresholds` 先 `Delete` 全部固定阈值行，再 `Create`）。如果院方已调整过现有指标（例如改了 MAP 档位），点"恢复默认"会把**所有**已调阈值冲掉，不仅仅是新增 5 个指标。

### 7.1 安全补种脚本（推荐，不破坏现有配置）

如果不想覆盖院方已调整的阈值，使用以下 SQL 只补种不存在的 5 个新指标：

```sql
INSERT INTO monitoring_threshold (tenant_id, metric_key, label, unit, scope, danger_low, warn_low, warn_high, danger_high, enabled, sort_order, created_at, updated_at)
VALUES
  (3, 'spo2',        '血氧饱和度', '%',     'global', 90,   95,    NULL, NULL, true, 60, now(), now()),
  (3, 'respiration', '呼吸',       '次/分', 'global', 8,    12,    20,   25,   true, 61, now(), now()),
  (3, 'tmp',         '跨膜压TMP', 'mmHg',  'global', NULL, NULL, 250,  350,  true, 62, now(), now()),
  (3, 'ap',          '动脉压AP',  'mmHg',  'global', NULL, NULL, 200,  250,  true, 63, now(), now()),
  (3, 'conductivity','电导率',     'mS/cm', 'global', 13.0, 13.5,  14.5, 15.0, true, 64, now(), now())
ON CONFLICT (tenant_id, metric_key, scope) DO NOTHING;
```

`ON CONFLICT DO NOTHING` 确保如果某个 key 已存在则不覆盖。`tenant_id` 值需按实际租户替换（ai-hms 老库默认 3）。

## 9. 风险与不确定项

- AP 按绝对值判断是合理默认，但不同机器符号可能不同，后续应让临床确认阈值。
- TMP、电导率默认阈值是经验值，上线后应允许院方在 admin 页调整。
- 如果 admin “恢复默认”会覆盖院方已调阈值，执行前需提醒用户先导出现有配置或手工补新增 5 行。
