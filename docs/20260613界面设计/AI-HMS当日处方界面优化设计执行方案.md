# AI-HMS 当日处方界面优化设计执行方案

适用页面：`ai-hms-frontend/src/pages/dialysis-processing/execution/TodayPrescription.tsx`  
上层页面：`ai-hms-frontend/src/pages/dialysis-processing/DialysisExecution.tsx`  
目标：不修改接口、不修改字段、不改变现有提取、保存、执行逻辑，只优化“当日处方”界面的样式、层级和临床核对流程。

---

## 1. 基于代码的当前实现

当前 `TodayPrescription.tsx` 已经具备完整的处方展示和维护能力：

1. 页面接收：
   - `patient`
   - `treatment`
   - `treatmentLoading`

2. 当前表单字段包括：
   - duration
   - dryWeight
   - extraWeight
   - dialysisMode
   - bloodFlow
   - substituteVolume
   - anticoagulant 相关字段
   - dialysate 相关字段
   - conductivity
   - temp
   - volume
   - notes

3. 页面会并行加载：
   - 患者治疗方案：`patientApi.getTreatmentPlan(patient.id)`
   - 患者处方列表：`prescriptionApi.list(patient.id)`

4. 页面会优先使用治疗方案回填表单，如果有当日处方或历史处方，会使用处方数据。

5. 当前已有功能：
   - 提取今日处方
   - 修改处方
   - 保存处方
   - 核对并执行
   - 展示透析材料清单
   - 展示动态调整记录
   - 底部 sticky 操作条

因此本次只做前端 UI 和层级优化，不动 API。

---

## 2. 当前界面主要问题

从截图和代码看，当前当日处方界面功能完整，但显示上还有几个问题：

### 2.1 患者信息重复

上层 `DialysisExecution.tsx` 已经在顶部渲染了 `PatientSummaryHeader`。当日处方组件内部又展示了一次患者姓名、状态、ID、性别年龄、费用类别、透析龄等信息，容易显得重复。

建议保留组件内的处方摘要，但弱化患者重复信息，重点展示“处方来源、方案、状态、更新时间、处方总数”。

---

### 2.2 核心处方参数分散

当前 5 个指标卡展示：

```text
透析方法
本次体重增加量
目标超滤量
透前血压
透析时间
```

下面又有：

```text
体重详情与基础标准
抗凝方案设定
透析液及通路设定
```

用户需要上下对照才能完成核对。建议把核对优先级提高：

```text
第一层：核心处方参数
第二层：抗凝方案
第三层：透析液/机器参数
第四层：耗材清单
第五层：动态调整记录
```

---

### 2.3 透析液参数太长

现在 `透析液及通路设定` 一次性展示很多字段，虽然字段齐全，但视觉上像一长排参数，阅读成本高。

建议分为三组：

```text
液体：透析液、液温、液量
浓度：Na、Ca、K、HCO3、葡萄糖
机器：电导度、血流、流速、置换量、备注
```

字段不变，只是展示分组。

---

### 2.4 材料清单和动态调整记录纵向堆叠

当前材料清单后面是动态调整记录，会继续拉长页面。建议在当日处方页面中，材料清单和动态调整记录可以并排展示：

```text
左侧：透析材料清单
右侧：处方动态调整记录
```

这样减少纵向滚动。

---

### 2.5 操作按钮可以更贴近“核对执行”流程

当前底部已经有 sticky 操作区，建议保留，但样式更明确：

```text
UPDATE: 时间
STATUS: 待执行

修改处方
核对并执行
```

如果处方未提取，则主按钮应引导“提取今日处方”。

## 4. 具体设计优化

### 4.1 顶部处方摘要

将当前患者信息区改为“处方摘要区”。

建议内容：

```text
支英俊  候诊
ID: 300410   女 / 72岁   费用: 市职工普通   透析龄: 待补充
来源方案：HF（干体重：0kg） · 处方总数：6

处方来源：治疗方案回填
干体重：0kg
治疗方式：HF
[提取今日处方]
```

注意：

1. 不删除患者信息，只降低权重。
2. 突出 `来源方案`、`处方总数`、`治疗方式`。
3. 提取今日处方按钮仍保留，但放在处方摘要右侧。

---

### 4.2 关键指标卡优化

保留当前 5 个指标，但样式更统一：

```text
透析方法 HF
本次体重增加量 0kg
目标超滤量 0L
透前血压 -- mmHg
透析时间 4h
```

建议：

1. 透析方法为主色卡。
2. 其他为白底卡。
3. 卡片高度控制在 88–96px。
4. 避免过高导致页面下方内容挤出首屏。

---

### 4.3 核心处方参数卡

新增一个“核心处方参数”大卡，优先展示需要核对的参数：

```text
治疗与容量：
透析方法
干体重
本次增量
目标超滤
透析时间

血流与通路：
血管通路
标准血流
透析液
流速
置换量
```

这些字段都来自现有 `metrics` 和 `form`，不需要新接口。

---

### 4.4 抗凝方案卡

抗凝方案单独放右侧，突出药剂和剂量：

```text
药剂名称：5%葡萄糖注射液
首剂量：1230
维持剂：--
维持量：--
注入速率：--
注入时间：--
```

建议：

1. 药剂名称用浅橙色背景。
2. 如果所有抗凝字段为空，显示“未配置抗凝方案”。
3. 字段仍然可编辑，沿用 `EditableField` 的 editing 控制。

---

### 4.5 透析液及机器设定分组

将当前长表单改为三组：

```text
液体：
透析液
液温
液量

浓度：
Na
Ca
K
HCO3
葡萄糖

机器参数：
电导度
血流
透析液流速
透析液分组
置换量
备注
```

字段不变，只调整视觉分组。

建议用 3 个浅灰小卡分组展示，减少“长列表参数”的感觉。

---

### 4.6 透析材料清单优化

当前表格字段保持：

```text
#
材料名称
分类
数量
编码
品牌
规格
备注
```

建议：

1. 表头 sticky。
2. 行高压缩到 36px 左右。
3. 材料名称加粗。
4. 如果无耗材，空状态提示：
   `当前处方和治疗方案均无耗材明细，可先检查治疗方案配置。`

---

### 4.7 动态调整记录优化

动态调整记录现在总是显示一个大表格，即使暂无数据也占空间。建议：

1. 和材料清单并排。
2. 暂无数据时显示紧凑空状态。
3. 增加提示：
   `修改处方后，建议记录调整内容、调整人和时间。`

后续如果有接口再接入真实调整记录，本次不新增接口。

---

### 4.8 底部固定操作条优化

保留当前 sticky 操作条，但强化主次按钮：

```text
UPDATE: 2026/5/31 22:55:55
STATUS: 待执行

[修改处方] [核对并执行]
```

编辑状态：

```text
[取消] [保存处方] [核对并执行]
```

如果 `!currentPrescription`：

```text
[提取今日处方]
```

如果 `!canExecute`，保持禁用，但提示文案更清楚：

```text
仅待执行处方可核对并执行
```

## 6. 可保留并增强的组件

当前已有组件可继续使用：

1. `MetricCard`
2. `EditableField`
3. `SectionCard`

建议轻量调整：

### 6.1 MetricCard

增加 `tone` 或 `compact`：

```tsx
function MetricCard({
  label,
  value,
  unit,
  primary,
  tone = 'blue',
  compact = true,
}: ...)
```

### 6.2 SectionCard

增加 `compact` 和 `description`：

```tsx
function SectionCard({
  title,
  icon,
  children,
  right,
  description,
  compact,
}: ...)
```

### 6.3 新增 ValueGroup

用于透析液分组展示：

```tsx
function ValueGroup({ title, children }: { title: string; children: ReactNode }) {
  return (
    <div className="rounded-xl border border-slate-200 bg-slate-50 p-4">
      <div className="mb-3 text-xs font-black text-slate-500">{title}</div>
      {children}
    </div>
  )
}
```
