## 2. 两系统表使用情况对照

### 2.1 v1.2 读写哪些表

**核心原则**：v1.2 有写操作的表全部使用 `Schedule_v2_*` 独立表，不共享写入任何当前系统表。仅 `TenantSetting` 和 `PlanChange` 列名完全一致且风险可控，保留共享。

| 表名 | v1.2操作 | 当前系统状态 | 列名冲突 | 处置方式 |
|------|----------|------------|---------|---------|
| `Schedule_Ward` | **读写** | ✅ 已有 | v1.2 需要 `ZoneType/ParentWardId/IsSubZone`，当前系统主表无这些列（在 `Schedule_WardExt`） | **← 冲突，需 v2 表**，从 Ward+WardExt 桥接 |
| `Schedule_Shift` | **读写** | ✅ 已有 | v1.2 需要 `ShiftCode`(NOT NULL)，当前系统无此列；且当前新增班次不写此列 | **← 冲突，需 v2 表**，从 Shift 桥接 |
| `Schedule_Machine` | **读写** | ❌ 不存在 | — | **新建 v2 表**，从 BedMachineExt 桥接数据 |
| `Schedule_MachineOutage` | **读写** | ✅ 已有 | v1.2: `MachineId` / 当前: `BedId` | **← 冲突，需 v2 表** |
| `Schedule_PatientShift` | **读写** | ✅ 已有，**核心冲突表** | 见 §2.2 详析 | **← 冲突，需 v2 表** |
| `Schedule_Calendar` | **读写** | ✅ 已有 | v1.2: `OpenWardIds text`, `OpenMachineIds text` / 当前: 关联表 `CalendarOpenWard`+`CalendarOpenBed`；v1.2 假日接口会写共享字段 | **← 冲突，需 v2 表** |
| `Schedule_PatientProfile` | **读写** | ✅ 已有 | v1.2: `FixedHdMachineId`+`FixedHdfMachineId` / 当前: `FixedHdBedId`+`FixedHdfBedId` | **← 冲突，需 v2 表** |
| `Schedule_ScheduleTemplate` | **读写** | ✅ 已有 | 列名一致，但关联新 v2 模板项表，独立隔离更安全 | **← 隔离，需 v2 表** |
| `Schedule_ScheduleTemplateItem` | **读写** | ✅ 已有 | v1.2: `FixedHdMachineId`+`FixedHdfMachineId` / 当前: `FixedHdBedId`+`FixedHdfBedId` | **← 冲突，需 v2 表** |
| `Schedule_ConflictQueue` | **写入** | ✅ 已有 | 列名一致，但 v1.2 写入会污染当前系统冲突列表 | **← 隔离，需 v2 表** |
| `Schedule_TenantSetting` | **只读** | ✅ 已有 | 列名一致 | **共享只读**（v1.2 不写，安全） |
| `Schedule_CrrtSession` | **读写** | ✅ 已有（GORM模型已定义） | v1.2: `MachineId` / 当前: `BedId` | **← 冲突，需 v2 表** |
| `Schedule_PlanChange` | **读写** | ✅ 已有（GORM模型已定义） | 列名一致 | **共享**（列名一致，v1.2只记变更记录） |
| `Schedule_Patient` | **读写** | ❌ 不存在 | — | **新建 v2 表**（v1.2 内部病人主档） |

### 2.2 核心冲突表：`Schedule_PatientShift`

这是两系统都必须完整读写的表，列名差异最大：

| 语义 | 当前系统列名 | v1.2列名 | 兼容性 |
|------|-------------|----------|--------|
| 租户ID | `TenantId` | `TenantId` | ✅ 一致 |
| 患者ID | `PatientId` | `PatientId` | ✅ 一致 |
| 排班日期 | **`TreatmentTime`** | **`ScheduleDate`** | ❌ 不同 |
| 班次ID | `ShiftId` | `ShiftId`(可空) | ⚠️ v1.2允许NULL |
| 病区ID | `WardId` | `WardId` | ✅ 一致 |
| 床位/机器ID | **`BedId`** | **`MachineId`** | ❌ 不同 |
| 状态 | `Status` | `Status` | ⚠️ 状态值可能不同 |
| 透析模式 | *(在PatientShiftExt)* | `DialysisMode` | ⚠️ 当前在扩展表 |
| 来源类型 | *(在PatientShiftExt)* | `SourceType` | ⚠️ 当前在扩展表 |
| 记录形态 | *(在PatientShiftExt)* | `RecordForm` | ⚠️ 当前在扩展表 |
| 确认时间 | *(在PatientShiftExt)* | `Confirm1At/By`等 | ⚠️ 当前在扩展表 |

**关键问题**: v1.2 使用 `ScheduleDate`（date类型）和 `MachineId`，当前系统使用 `TreatmentTime`（timestamp类型）和 `BedId`。这是**同一个表的两个不同列**，会导致两系统读写不同字段，产生数据不一致。

### 2.3 列名/结构不兼容表（均需独立 v2 表）

以下表 v1.2 有写操作但列名或结构与当前系统不兼容：

| 表名 | 当前系统列/结构 | v1.2 列/结构 | 冲突性质 |
|------|----------|---------|---------|
| `Schedule_Ward` | 主表无 `ZoneType/ParentWardId/IsSubZone`，在 `Schedule_WardExt` | 模型直接包含 `ZoneType/ParentWardId/IsSubZone` | **列不在主表** |
| `Schedule_Shift` | 当前无 `ShiftCode`，新增班次不写此列 | `ShiftCode` (NOT NULL) | **缺列**，当前 INSERT 失败 |
| `Schedule_MachineOutage` | `BedId` (NOT NULL, `schedule_ext.go:173`) | `MachineId` (NOT NULL, `models.go:57`) | **列名不同** |
| `Schedule_CrrtSession` | `BedId` (NOT NULL, `schedule_ext.go:230`) | `MachineId` (NOT NULL, `models.go:165`) | **列名不同** |
| `Schedule_PatientProfile` | `FixedHdBedId`, `FixedHdfBedId` (`schedule_ext.go:60-61`) | `FixedHdMachineId`, `FixedHdfMachineId` (`models.go:108-109`) | **列名不同** |
| `Schedule_ScheduleTemplateItem` | `FixedHdBedId`, `FixedHdfBedId` (`schedule_ext.go:129-130`) | `FixedHdMachineId`, `FixedHdfMachineId` (`models.go:191-192`) | **列名不同** |

### 2.4 语义冲突表（列名一致但仍需独立 v2 表）

以下表列名一致，但 v1.2 的读写操作会直接触碰当前系统数据，影响现有功能：

| 表名 | 冲突原因 |
|------|---------|
| `Schedule_Calendar` | v1.2 假日接口写 `IsDialysisDay/HolidayMode/Note` 等字段，会改动当前系统日历语义（§4.1.3 补建 v2 表 + 桥接） |
| `Schedule_ConflictQueue` | v1.2 冲突检测写入同一表，当前系统 `/schedule` 页面会显示 v1.2 的冲突项，干扰用户判断 |
| `Schedule_ScheduleTemplate` | 虽然列名一致，但 v1.2 的模板管理（`/admin/templates/rebuild`）写入共享表会覆盖当前系统模板数据 |

## 3. 迁移策略：完全隔离（方案A 增强版）

**核心原则**：v1.2 有写操作的表**全部**使用独立 `Schedule_v2_*` 表，与当前系统的 `Schedule_*` 表物理隔离。

```
两系统共享同一 PostgreSQL database: dialysis
        但读写不同表，互不干扰

  当前系统 (:8080)        v1.2 (:8081)
  ┌──────────────┐       ┌──────────────────┐
  │ Schedule_Ward│       │Schedule_v2_Ward  │ ← 隔离
  │ Schedule_Shift│      │Schedule_v2_Shift │ ← 隔离
  │ Schedule_Bed │       │Schedule_Machine  │ ← 新建
  │ Schedule_PatientShift│← 不碰 →│Schedule_v2_PatientShift│ ← 隔离
  │ Schedule_Calendar│   │Schedule_v2_Calendar│ ← 隔离
  │ Schedule_ConflictQueue│← 不碰 →│Schedule_v2_ConflictQueue│ ← 隔离
  │ Schedule_TenantSetting│← 只读 ─│ 共享只读          │ ✓ 安全
  │ Schedule_PlanChange│  │ 共享读写          │ ✓ 列名一致
  └──────────────┘       └──────────────────┘
```

优点：
- **零影响现有功能**：当前系统的表结构不发生任何变更（无 ALTER）
- v1.2 模型列名完全不变
- 两系统物理隔离，不会相互污染数据

缺点：
- 需要对 v1.2 代码做 11 个模型的表名映射（将 TableName 从 `Schedule_X` 改为 `Schedule_v2_X`）
- 数据不互通（在 v1.2 创建的排班不出现在当前系统排班管理页面）
- 需要从当前系统表桥接基础数据到 v2 表（一次性）

#### 4.1.2 数据桥接：从当前系统表一次性导入基础数据（3 项）

```sql
-- 桥接1：病区（Schedule_Ward + Schedule_WardExt → Schedule_v2_Ward）
INSERT INTO "Schedule_v2_Ward" (
  "Id", "TenantId", "Name", "ZoneType", "ParentWardId", "IsSubZone",
  "Sort", "IsDisabled", "Note", "CreatorId", "CreateTime", "LastModifyTime"
)
SELECT
  w."Id", w."TenantId", w."Name",
  COALESCE(e."ZoneType", 'A'),
  e."ParentWardId",
  COALESCE(e."IsSubZone", FALSE),
  COALESCE(w."Sort", 0),
  COALESCE(w."IsDisabled", FALSE),
  w."Note",
  w."CreatorId",
  w."CreateTime",
  w."LastModifyTime"
FROM "Schedule_Ward" w
LEFT JOIN "Schedule_WardExt" e ON e."WardId" = w."Id" AND e."TenantId" = w."TenantId"
WHERE NOT EXISTS (
  SELECT 1 FROM "Schedule_v2_Ward" v2 WHERE v2."Id" = w."Id" AND v2."TenantId" = w."TenantId"
);

-- 桥接2：班次（Schedule_Shift → Schedule_v2_Shift）
INSERT INTO "Schedule_v2_Shift" (
  "Id", "TenantId", "Name", "ShiftCode", "StartTime", "EndTime",
  "Sort", "IsDisabled", "CreatorId", "CreateTime", "LastModifyTime"
)
SELECT
  "Id", "TenantId", "Name",
  CASE
    WHEN "Sort" = 1 THEN 'MORNING'
    WHEN "Sort" = 2 THEN 'AFTERNOON'
    WHEN "Sort" = 3 THEN 'NIGHT'
    ELSE 'OTHER'
  END,
  "StartTime", "EndTime",
  "Sort", "IsDisabled",
  "CreatorId", "CreateTime", "LastModifyTime"
FROM "Schedule_Shift"
WHERE NOT EXISTS (
  SELECT 1 FROM "Schedule_v2_Shift" v2 WHERE v2."Id" = "Schedule_Shift"."Id" AND v2."TenantId" = "Schedule_Shift"."TenantId"
);

-- 桥接3：机器（Schedule_BedMachineExt + Schedule_Bed → Schedule_Machine）
INSERT INTO "Schedule_Machine" (
  "TenantId", "WardId", "Code", "Name", "MachineType",
  "PositionIndex", "IsDisabled", "Sort", "LegacyBedId"
)
SELECT
  e."TenantId",
  COALESCE(b."WardId", 0),
  e."MachineCode",
  e."LegacyBedName",
  e."MachineType",
  e."PositionIndex",
  e."IsDisabled",
  e."PositionIndex",
  e."BedId"
FROM "Schedule_BedMachineExt" e
JOIN "Schedule_Bed" b ON b."Id" = e."BedId" AND b."TenantId" = e."TenantId"
WHERE NOT EXISTS (
  SELECT 1 FROM "Schedule_Machine" m
  WHERE m."LegacyBedId" = e."BedId" AND m."TenantId" = e."TenantId"
);
```

> **注意**：以上为一次性桥接。后续 v1.2 通过其 `/admin` 接口维护 v2 表数据，当前系统继续在原表维护。两套数据独立维护。

### 4.2 v1.2 Go 代码改造（8 项）

#### 改造0：GORM 命名策略——添加大小写敏感配置（P0 必须先做）

文件: `backend/internal/db/db.go` → 函数 `Open()`

v1.2 的 `Open()` 仅有 `gorm.Open(postgres.Open(dsn), &gorm.Config{})`，**未设置** `SingularTable` 和 `NoLowerCase`。老库 PostgreSQL 表名/列名是双引号包围的大写敏感形式（如 `"Schedule_Shift"`、`"TenantId"`），GORM 默认小写化会导致所有查询失败。

```go
// import 新增
import "gorm.io/gorm/schema"

// Open() 函数改为
func Open(dsn string) (*gorm.DB, error) {
    gormConfig := &gorm.Config{
        NamingStrategy: schema.NamingStrategy{
            SingularTable: true,  // 禁用复数表名
            NoLowerCase:   true,  // 保持大小写，与数据库双引号列名一致
        },
        DisableForeignKeyConstraintWhenMigrating: true,
    }
    g, err := gorm.Open(postgres.Open(dsn), gormConfig)
    // ... 其余连接池代码不变
}
```

#### 改造1：禁用 AutoMigrate + 唯一索引用 v2 表名（P0 安全）

文件: `backend/internal/db/db.go`

```go
// 改造前

    if err := g.AutoMigrate(model.AllModels()...); err != nil {  // ← 删除此行
        return err
    }
    return createUniqueIndexes(g)

func Migrate(g *gorm.DB) error {
```

#### 改造3：响应格式统一包装

新增文件: `backend/internal/api/wrapper.go`

```go
package api

import (
    "net/http"
    "time"
    "github.com/gin-gonic/gin"
)

// ok 统一成功响应: {success:true, data:..., timestamp:...}
func ok(c *gin.Context, v interface{}) {
    c.JSON(http.StatusOK, gin.H{"success": true, "data": v, "timestamp": time.Now().Format(time.RFC3339)})
}

// fail 统一错误响应: {success:false, error:..., timestamp:...}
func fail(c *gin.Context, code int, msg string) {
    c.AbortWithStatusJSON(code, gin.H{"success": false, "error": msg, "timestamp": time.Now().Format(time.RFC3339)})
}
```

然后将所有 handler 中的：
- `c.JSON(http.StatusOK, res)` → `ok(c, res)`
- `c.JSON(http.StatusBadRequest, gin.H{"error": ...})` → `fail(c, 400, msg)`
- `c.JSON(http.StatusNotFound, gin.H{"error": ...})` → `fail(c, 404, msg)`
- `c.JSON(http.StatusConflict, gin.H{"error": ...})` → `fail(c, 409, msg)`

保留 `failInternal` 做服务端脱敏（日志记录真实错误，返回通用提示）。

#### 改造4：数据桥接——repo 层添加自动种子

新增文件: `backend/internal/repo/machine_bridge.go`

```go
package repo

import (
    "gorm.io/gorm"
    "github.com/sdsph/dialysis-scheduling/internal/model"
)

// EnsureMachinesSeeded 若 Schedule_Machine 为空，从 BedMachineExt 自动桥接。
func EnsureMachinesSeeded(g *gorm.DB) error {
    var count int64
    if err := g.Model(&model.Machine{}).Count(&count).Error; err != nil {
        return err
    }
    if count > 0 {
        return nil
    }
    return g.Exec(`
        INSERT INTO "Schedule_Machine" ("TenantId","WardId","Code","Name","MachineType","PositionIndex","IsDisabled","Sort","LegacyBedId")
        SELECT e."TenantId", b."WardId", e."MachineCode", e."LegacyBedName", e."MachineType",
               e."PositionIndex", e."IsDisabled", e."PositionIndex", e."BedId"
        FROM "Schedule_BedMachineExt" e
        JOIN "Schedule_Bed" b ON b."Id" = e."BedId" AND b."TenantId" = e."TenantId"
        WHERE NOT EXISTS (SELECT 1 FROM "Schedule_Machine" m WHERE m."LegacyBedId" = e."BedId" AND m."TenantId" = e."TenantId")
    `).Error
}
```

#### 改造5：鉴权适配——在 :8080 侧加反向代理路由

**方案**：在当前 `ai-hms-backend` 中增加反向代理路由组 `/api/v2/*`，复用现有 JWT 鉴权中间件，解析出 tenant/role/user 后反向代理到 `:8081`。这样 iframe 内的前端 API 请求从 `:8080/api/v2/...` 走，由当前系统中间件统一鉴权。

在当前 `ai-hms-backend` 新增文件：`internal/api/v1/smart_schedule_proxy.go`

```go
package v1

import (
    "net/http/httputil"
    "net/url"
    "github.com/gin-gonic/gin"
)

var smartScheduleProxy = func() *httputil.ReverseProxy {
    u, _ := url.Parse("http://localhost:8081")
    return httputil.NewSingleHostReverseProxy(u)
}()

func ProxyToSmartSchedule(c *gin.Context) {
    // 从当前系统 JWT 上下文提取 tenant/role/user，注入到请求头
    c.Request.Header.Set("X-Tenant-Id", tenantIDFromCtx(c))
    c.Request.Header.Set("X-Role", roleFromCtx(c))
    c.Request.Header.Set("X-User-Id", userIDFromCtx(c))
    // 路径转换: /api/v2/schedule/week → /api/v1/schedule/week
    c.Request.URL.Path = "/api/v1" + c.Param("path")
    smartScheduleProxy.ServeHTTP(c.Writer, c.Request)
}
```

主路由注册（`internal/api/v1/router.go` 或等效文件）：
```go
v2 := router.Group("/api/v2")
v2.Use(authMiddleware) // 复用当前系统 JWT 鉴权
v2.Any("/*path", ProxyToSmartSchedule)
```

iframe 页面的 API 调用改为指向 `/api/v2/...`（同源，无跨域问题）。
