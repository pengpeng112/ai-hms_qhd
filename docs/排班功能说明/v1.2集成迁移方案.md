### 2.2 核心冲突表：`Schedule_PatientShift`

这是两系统都必须完整读写的表，列名差异最大：

| 语义 | 当前系统列名 | v1.2列名 | 兼容性 |
|------|-------------|----------|--------|
| 租户ID | `TenantId` | `TenantId` | ✅ 一致 |
| 患者ID | `PatientId` | `PatientId` | ✅ 一致 |
| 排班日期 | **`TreatmentTime`** | **`ScheduleDate`** | ❌ 不同 |
| 班次ID | `ShiftId` | `ShiftId`(可空) | ⚠️ v1.2允许NULL |
| 病区ID | `WardId` | `WardId` | ✅ 一致 |
| 床位/机器ID | **`BedId`** | **`MachineId`** | ❌ 不同 |
| 状态 | `Status` | `Status` | ⚠️ 状态值可能不同 |
| 透析模式 | *(在PatientShiftExt)* | `DialysisMode` | ⚠️ 当前在扩展表 |
| 来源类型 | *(在PatientShiftExt)* | `SourceType` | ⚠️ 当前在扩展表 |
| 记录形态 | *(在PatientShiftExt)* | `RecordForm` | ⚠️ 当前在扩展表 |
| 确认时间 | *(在PatientShiftExt)* | `Confirm1At/By`等 | ⚠️ 当前在扩展表 |

**关键问题**: v1.2 使用 `ScheduleDate`（date类型）和 `MachineId`，当前系统使用 `TreatmentTime`（timestamp类型）和 `BedId`。这是**同一个表的两个不同列**，会导致两系统读写不同字段，产生数据不一致。

### 2.3 `Schedule_PatientProfile` 列名差异

| 语义 | 当前列名 | v1.2列名 |
|------|---------|----------|
| 固定HD床位 | **`FixedHdBedId`** | **`FixedHdMachineId`** |
| 固定HDF床位 | **`FixedHdfBedId`** | **`FixedHdfMachineId`** |

## 3. 迁移策略：方案选择

### 方案A：新建独立表（推荐 ✅）

**为 v1.2 创建全新的 `Schedule_v2_PatientShift` 等独立表**，与原表互不干扰。

优点：
- 零影响现有功能
- 不需要改 v1.2 模型列名
- 两系统完全隔离

缺点：
- 需要对 v1.2 代码做表名映射（将 model 中的 TableName() 从 `Schedule_PatientShift` 改为 `Schedule_v2_PatientShift`）
- 数据不互通（在 v1.2 创建的排班不会出现在当前系统的排班管理页面）

### 方案B：共享表 + v1.2列映射（成本可控 ✅）

**让 v1.2 共享现有表，但在 Go model 层面做列名映射，不改数据库**。

优点：
- 数据互通
- 不需要新表

缺点：
- 列名无法直接映射：`ScheduleDate`→`TreatmentTime`，`MachineId`→`BedId` 需要在 v1.2 侧做 GORM column tag 映射
- `DialysisMode`/`SourceType`/`RecordForm` 在当前系统中在 `PatientShiftExt` 表，v1.2 在 `PatientShift` 主表
- 需要额外添加 `Schedule_Machine` 新表

### 推荐：方案A（独立表）+ 共享只读表

v1.2 的写操作只写到新建的独立表，读操作从共享表 + 新表联合读取。

### 4.2 v1.2 Go 代码改造

#### 改造1：禁用 AutoMigrate（P0 安全风险）

文件: `backend/internal/db/db.go`

```go
// 改造前
func Migrate(g *gorm.DB) error {
    if err := g.AutoMigrate(model.AllModels()...); err != nil {  // ← 删除此行及相关
        return err
    }
    return createUniqueIndexes(g)
}

// 改造后
func Migrate(g *gorm.DB) error {
    return createUniqueIndexes(g) // 只建索引，不改表结构
}
```

#### 改造2：PatientShift 表名映射

文件: `backend/internal/model/models.go:159`

```go
// v1.2 原有
func (PatientShift) TableName() string { return "Schedule_PatientShift" }

// 改为独立表
func (PatientShift) TableName() string { return "Schedule_v2_PatientShift" }
```

#### 改造3：数据桥接——从 BedMachineExt 读机器信息

v1.2 的 repo.LoadBoard 需要从 `Schedule_Machine` 表读机器。但我们没有这个表的数据。需要在 v1.2 启动时从 `Schedule_BedMachineExt` 桥接或预先导入数据。

新增文件: `backend/internal/repo/machine_bridge.go`

```go
// 从 Schedule_BedMachineExt 桥接到 Machine 视图
// 如果 Schedule_Machine 为空，从 BedMachineExt 回退读取
```

#### 改造4：响应格式包装

v1.2 直接返回裸 JSON。当前前端期望 `{success: true, data: ...}`。

在 `api.go` 的 handler 返回处统一包装：

```go
// 当前
c.JSON(http.StatusOK, res)

// 改为
c.JSON(http.StatusOK, gin.H{"success": true, "data": res, "timestamp": time.Now().Format(time.RFC3339)})
```

#### 改造5：鉴权适配

v1.2 使用 X-Role/X-Tenant-Id 请求头。当前系统使用 JWT Bearer Token。

方案：在 v1.2 前端 nginx 或 vite proxy 层做转换：
```
前端请求 → proxy中间件 → 解析JWT → 设置 X-Role/X-User-Id/X-Tenant-Id → 转发到v1.2
```

或在当前系统的后端添加一个简单的代理路由。

## 7. 预期效果

改造完成后：
- 当前排班管理页面 (`/schedule`) **功能完全不变**
- 新增菜单"智能排班" (`/smart-schedule`)，进入 v1.2 的独立界面
- v1.2 的数据写入独立表 `Schedule_v2_*`，不污染现有表
- v1.2 的生成算法直接复用（比当前系统自己实现的 engine 更完整，多了 CRRT/临时透析/停机/假日/方案变更）
- 用户可在两个窗口之间切换，互不影响

## 8. 风险评估汇总
