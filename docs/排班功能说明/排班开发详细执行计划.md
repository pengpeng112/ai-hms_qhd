# 排班系统开发详细执行计划

> 适用项目：`ai-hms_qhd`
>
> 目标：按照《透析排班子程序规则规范_v1(1).md》和参考后端 `透析排班-backend-v1.1(3).zip` 的能力，将当前系统从“手工排班 + 单日模板应用”升级为“规则驱动的 2/4 周草稿生成 + 人工确认 + 扰动处理 + 冲突队列”的完整排班子系统。

---

## 0. 当前系统基线

### 已有能力

1. 排班周视图/日视图：`ai-hms-frontend/src/pages/Schedule.tsx`
2. 手工新建排班：`CreateScheduleModal.tsx` + `POST /api/v1/patient-shifts`
3. 排班修改/取消/换床/拖拽交换：`Schedule.tsx` + `useScheduleDragDrop.ts`
4. 排班模板基础能力：模板列表、编辑、应用
5. 排班配置扩展：`WardExt`、`BedMachineExt`、`PatientProfile`、`TenantSetting`、`Calendar`、`MachineOutage`
6. Board 预检部分能力：机器模式、Profile、CRRT 配置、非透析日、停机、冲突检测

### 当前主要缺口

1. 没有 2/4 周自动草稿生成算法
2. 没有 HD/HDF 两轮分配算法
3. 没有 HDF 奇偶周自动错峰
4. 没有固定机位/就近机位/新病人初始分配
5. 没有排满顺延建议
6. 没有持久化冲突队列
7. 没有三级确认工作流
8. 没有正式请假/缺席/补透/临时透析流程
9. 没有 CRRT 时间段排班
10. 没有设备停机迁移和假日挪班
11. 状态机和来源字段语义仍需统一

## 1. 总体开发原则

### 1.1 不要改动老库结构的硬规则

当前后端连接旧 PostgreSQL，严禁启用 AutoMigrate、DropTables 或任意自动 DDL。

开发如需新表，必须：

1. 明确 SQL 脚本放入 `docs/sql/`
2. 人工执行 SQL
3. Go 模型只映射已存在表
4. 所有表名/字段名保持双引号大小写敏感

### 1.2 算法原则

排班算法必须遵循：

1. 主方案 → 备选方案 → 报警人工裁决
2. 算法只读患者骨架，不自动决定患者分区/频率/班次/治疗模式
3. 排不开时不得强行乱排，必须进入冲突队列
4. 生成结果默认是草稿，不直接生效

## 2. 第一阶段：统一状态机与基础字段

### 2.1 统一排班状态机

规范状态：

| 状态码 | 含义 |
|---|---|
| 0 | 待排 |
| 10 | 草稿 |
| 20 | 已确认 |
| 50 | 透析中 |
| 60 | 已完成 |
| 70 | 已取消 |
| 80 | 缺席 |

当前系统存在 legacy/new 状态映射，需要先整理。

#### 需要修改/核查文件

1. `ai-hms-backend/internal/models/patient_shift.go`
2. `ai-hms-backend/internal/services/patient_shift_service.go`
3. `ai-hms-backend/internal/services/schedule_week_service.go`
4. `ai-hms-frontend/src/pages/Schedule.tsx`
5. `ai-hms-frontend/src/services/restClient.ts`

#### 执行任务

1. 梳理当前 `MapPatientShiftStatusNewToLegacy` 和 `MapPatientShiftStatusLegacyToNew`
2. 新增文档 `docs/schedule-status-mapping.md`
3. 明确手工创建默认状态：建议 `20 已确认`，算法/模板生成默认 `10 草稿`
4. 周视图 `statusName` 统一按新状态显示
5. 取消操作不再物理删除，改成状态 `70 已取消`
6. 缺席新增状态 `80 缺席`

#### 验收标准

1. 周视图能显示草稿/已确认/已取消/缺席
2. 已取消和缺席不参与床位冲突占用
3. 已完成/透析中/历史日期不可编辑

### 2.2 统一 sourceType / recordForm

建议语义：

| 字段 | 值 | 含义 |
|---|---|---|
| SourceType | 10 | 常规模板/算法生成 |
| SourceType | 20 | 临时透析 |
| SourceType | 30 | 手工创建 |
| RecordForm | 10 | 规律透析 |
| RecordForm | 20 | CRRT |

#### 执行任务

1. 周视图优先读取 `Schedule_PatientShiftExt.SourceType`
2. 手工创建排班时同步写入 `Schedule_PatientShiftExt`，`SourceType=30`，`RecordForm=10`
3. 模板/算法生成排班时写入 `SourceType=10`
4. 临时透析写入 `SourceType=20`
5. CRRT 写入 `RecordForm=20`

#### 验收标准

1. 手工排班显示“手工”或无角标
2. 模板/算法生成显示“模板/规则”
3. 临时透析显示“临”
4. CRRT 不混入普通三班网格

### 4.2 频率规则

```go

```

频率：

| 编码 | 含义 | 日期 |
|---|---|---|
| 10 | 一三五 | 周一/三/五 |
| 20 | 二四六 | 周二/四/六 |
| 30 | 每周两次 | 周二/四 |
| 40 | 每周一次 | 周四 |
| 90 | 临时 | 不参与规律生成 |

### 4.3 生成接口

新增：

```http
POST /api/v1/schedule/generate
```

请求：

```json
{
  "templateId": 1,
  "startDate": "2026-06-08",
  "weeks": 2,
  "wardId": 3
}
```

响应：

```json
{
  "startDate": "2026-06-08",
  "weeks": 2,
  "dialysisDays": 12,
  "drafts": 86,
  "conflicts": 4,
  "parityAssigned": 3
}
```

### 4.4 生成流程

1. 读取模板项
2. 读取 Board 快照
3. 展开未来 2/4 周透析日
4. 按 `(WardId, ShiftId, Date)` 分组
5. 过滤 `FreqPattern=90`
6. 对每个应排 session 执行机位分配
7. 生成 `Status=10 草稿`
8. 写 `PatientShiftExt`
9. 写冲突队列
10. 返回摘要

### 验收标准

1. 可生成 2 周草稿
2. 可生成 4 周草稿
3. 已有有效排班不重复生成
4. 已取消/缺席不被自动复活
5. 无可用机位进入冲突队列

## 5. 第四阶段：HD/HDF 两轮分配算法

### 5.1 两轮分配

规则：

1. 第一轮：HDF 次优先占 HDF 机
2. 第二轮：HD 次优先占 HD 机
3. HD 机满后可溢出到 HDF 空位
4. HDF 不可排入 HD 机

### 5.2 固定机位

顺序：

HD：

1. `FixedHdBedId`
2. 同区空闲 HD 机
3. 同区空闲 HDF 机
4. 顺延或冲突

HDF：

1. `FixedHdfBedId`
2. 同区最近 HDF 机
3. 冲突队列

### 5.3 候选评分

实现：

```go
func pickBest(candidates []BedCandidate, session SessionItem, cell Cell) BedCandidate
```

评分：

1. 靠近固定机位加分
2. 相邻机位已占用加分，保护整片空机
3. 同模式/同方案病人相邻加分
4. 不做负载均衡

### 验收标准

1. HDF 患者优先 HDF 机
2. HD 优先 HD，满后才用 HDF
3. 固定机位可用时优先固定机位
4. 固定机位不可用时就近

## 6. 第五阶段：HDF 奇偶周错峰

### 6.1 奇偶周锚点

默认：`2025-01-06`

配置项：

```txt
Schedule_TenantSetting.OddEvenWeekAnchorMonday
```

### 6.2 自动分配 HDF 奇偶周

规则：

1. 已有 `HdfWeekParity` 保持不变
2. 新 HDF 患者按 HDF weekday 分到负载较轻的一侧
3. 写回 `PatientProfile` 和 `ScheduleTemplateItem`

### 6.3 HDF 替换语义

实现：

```go
func DecideMode(anchor time.Time, hdfEnabled bool, hdfWeekday *int16, hdfWeekParity *int16, date time.Time) string
```

规则：

1. 非 HDF 患者：HD
2. 日期不是 HDF 日：HD
3. 日期是 HDF 日且奇偶命中：HDF
4. 否则 HD

### 验收标准

1. HDF 不增加总次数，只替换某次 HD
2. 奇偶周跨年不漂移
3. 新 HDF 患者自动错峰

## 7. 第六阶段：排满顺延

### 7.1 顺延逻辑

规则：

1. 不加床
2. 先找同日后续班次
3. 再找后续透析日原班次
4. 再找后续透析日其他班次
5. 找到则生成建议草稿 + 冲突提示
6. 找不到则报警

### 7.2 实现函数

```go
func spillToLaterSlot(session SessionItem, cell Cell) *SuggestedShift
func laterSlots(cell Cell) []Cell
```

### 验收标准

1. 当前班次满时不会强塞
2. 能顺延到后续空位
3. 顺延必须写冲突队列提示人工确认

```http

```

功能：

1. 状态改 `70 已取消`
2. 写取消原因

4. 可选生成补透建议

### 9.2 缺席

接口：

```http
POST /api/v1/patient-shifts/:id/absent
```

功能：

1. 状态改 `80 缺席`
2. 留痕
3. 机位可被临时透析借用

### 9.3 临时透析

接口：

```http
POST /api/v1/schedule/temporary
```

请求：

```json
{
  "patientId": 1001,
  "wardId": 3,
  "date": "2026-06-08",
  "dialysisMode": "HD"
}
```

规则：

1. 当前/指定班次开始找空位
2. 同区优先
3. 模式匹配
4. 可借请假/缺席机位
5. 不挤规律病人
6. 无空位写冲突队列

### 9.4 补透建议

接口：

```http
POST /api/v1/schedule/makeup/suggest
```

功能：

1. 根据差异检测找到少排患者
2. 查未来空位
3. 写 `MAKEUP_SUGGEST`
4. 人工确认后创建排班

## 10. 第九阶段：设备停机与假日

### 10.1 设备停机

接口：

```http
POST /api/v1/schedule/outages
```


规则：

2. 已有排班受影响时取消
3. 给前后 7 天挪班建议

## 17. 关键风险提醒

1. 不要直接照搬参考后端的 `MachineId`，当前系统应使用 `BedId + BedMachineExt` 表达机器位。
2. 不要把现有“应用模板”误认为规范里的“2/4 周模板复制生成”。
3. 没有冲突队列就不要开发自动生成算法，否则排不开时无法闭环。
4. 状态机不统一前，不要开发确认流，否则后续会返工。
5. CRRT 不能塞入普通三班网格，必须按时间段单独建模。
6. 请假/缺席不能物理删除，必须留痕。
7. 方案变更不能直接改已确认近期排班，必须按确认层级判断。
