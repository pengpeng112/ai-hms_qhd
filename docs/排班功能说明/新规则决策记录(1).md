# 新 AI 透析排班子程序 — 规则决策记录

> 与刘子栋医生逐条讨论确定。每条含:问题、结论、对代码/数据模型的含义。

---

## 决策 1:HDF 的含义 = 替换,非额外

**问题**:"每两周一次 HDF" 是替换常规某一次 HD,还是在原频率外额外加一次?

**结论**:**替换**。病人透析总次数不变(由频率医嘱决定),每 2 周把其中**某一次的模式从 HD 换成 HDF**。

**含义**:
- "治疗模式(HD / HDF / CRRT)" 是 **每一条排班记录(每一次透析)** 的属性,不是病人的固定属性。
- 同一病人不同次的排班可以是不同模式。
- 数据模型:`PatientShift.DialysisMode` 按次存储。

---

## 决策 2:HDF 那一次的机位 = 双固定优先,退而求其次就近,再冲突则报警

**问题**:做 HDF 那一次,病人去哪台 HDF 机?对 HDF 机也固定吗?

**结论**(优先级链):
1. **方案 A(首选)**:**双固定**——病人平时固定一台 HD 机,做 HDF 时固定回到**同一台 HDF 机**。
2. **方案 B(退路)**:方案 A 排不开(固定 HDF 机被占)时,**就近**调度一台空闲 HDF 机。
3. **报警**:方案 B 仍有矛盾(无空闲 HDF 机)时,**发出报警,交排班者人工裁决**。

**含义**:
- 病人需要两个"归属机位"字段:固定 HD 机、固定 HDF 机。
- HDF 机位也要尽量为该病人预留/记忆。

---

## ★ 通用设计原则(贯穿所有规则)

**主方案 → 备选方案 → 仍冲突则报警人工裁决。**
排班算法不追求"自动解决一切",冲突无法自动满足时,宁可**报警让人工决定**,也不强行乱排。
→ 系统需要一套"冲突/报警"机制和待处理队列。

---

## 决策 3:HDF 日 = 固定星期几,第几周由系统按均匀性测算

**问题**:"每两周一次 HDF" 具体落在哪一次?固定吗?谁定?

**结论**:
1. **星期几:固定**。每个病人的 HDF 固定在某个星期几(且必须是该病人已有的透析日——因为 HDF 是替换某次 HD,不能凭空加一天)。
2. **第 1 周 or 第 2 周:由系统测算**。系统按"HDF 机位在各周的均匀性/利用率"来决定放奇数周还是偶数周,**错峰**避免所有 HDF 病人挤同一周把 HDF 机挤爆。
3. **尽量固定**:一旦排定就尽量保持;**只有 HDF 机位实在不够时才微调时间**(再不行→报警,见通用原则)。

**含义**:
- 病人属性:HDF 星期几(取自其透析日之一)、HDF 所属周(奇/偶,系统算)。
- 系统需要一个"HDF 机位跨周负载均衡"的测算步骤。
- 待澄清:HDF 具体星期几是医嘱指定,还是系统在病人透析日里自动挑?(下个问题或后续确认)

---

## 决策 4:HD 病人优先用 HD 机,HD 机满了才借用空闲 HDF 机位

**问题**:HDF 机空位允不允许 HD 病人用?

**结论**:
- HD 病人**优先安排 HD 机**;
- **HD 机全满后**,才把 HD 病人安排到空闲的 HDF 机位上;
- (HDF 病人始终优先占用 HDF 机,见决策 2/3)。

**含义——确立排班算法的"两轮分配"骨架**:
1. **第一轮(HDF 次)**:把所有"该做 HDF 的次"排进 HDF 机(双固定优先)。
2. **第二轮(HD 次)**:HD 次优先排 HD 机;HD 机满 → 溢出到 HDF 机剩余空位。
- 这样 HDF 机既有保障(HDF 病人先占),又不浪费(空位可被 HD 借用)。

---

## 决策 5:分区来自病人输入标签;门诊+传染病=拒绝收治(可权限override进C区)

**问题**:病人分到哪个区怎么定?C 区怎么判定?院感和门诊/住院谁优先?

**结论**:
- **分区是病人的输入属性,系统读取而非计算**:门诊→A、住院→B、C 区病人**在输入时即带 C 区标签**。排班只在该区内部选机器/床位/班次。
- **门诊病人若有传染病 → 拒绝收治(reject)**,不予排班。
- **例外(权限 override)**:医生或护士长(按权限)可把该病人**标识为 C 区特征病人**,从而安排进 C 区。

**含义**:
- 排班系统要有 **"拒绝收治"** 这一种结果(不是所有病人都能排进去)。
- 要有 **权限控制的人工 override** 机制(医生/护士长可越过默认规则,指定病人进 C 区)。
- 病人输入属性需含:门诊/住院、C 区标签、院感(传染病)状态。

**待澄清**:住院病人若有传染病怎么处理?是否有独立"传染病隔离区",还是并入 C 区?(下一问)

---

## 决策 6:不收确诊传染病病人;急诊人道救治可进 C 区(无独立传染病区)

**问题**:传染病病人放哪?要不要按病种隔离?要不要传染病区?

**结论**:
- 本院**不接收明确传染病病人**(确诊传染病一律拒收)。
- **例外**:出于人道主义的**急诊救治**,可安排在 **C 区**。
- 因此:**不设独立传染病区,也不做按病种隔离子区**。老系统的"传染病病区/院感隔离"逻辑在新系统中**取消**。

**含义**:
- 分区模型简化为 A / B / C 三类(+ 可扩展子区),无传染病专区。
- 院感隔离不再是排班算法的常规约束;只在 C 区内部消化急诊传染病例。
- C 区职能汇总:危重、新病人、**无明确院感指标者**、急症透析、传染病急诊人道救治。
- "无明确院感指标"=指标未查清前先进 C 区(预防性),查清阴性后再转出。

---

## 决策 7:C 区转 A/B 区,须医生先明确改病人标签,系统方可据此排班

**问题**:C 区病人何时、怎么转回 A/B?谁触发?

**结论**:
- C 区病人要改到 A 区或 B 区,**必须由医生明确变更病人的分区标签**;
- **只有标签改了之后**,系统才允许把该病人排进 A/B 区。
- 系统**不自动转区**,只跟随标签。

**含义(确立一条核心原则)**:
- **排班系统从不修改病人的分区归属,只读取标签**;分区标签的变更是**医生的权限动作**。
- 配合决策 5/6:分区(含 C↔A/B 流转)完全由"病人标签 + 权限人工操作"驱动,排班算法只在标签限定的区内工作。
- (可选增强:系统可提示"院感结果已回阴性,建议考虑转出",但绑定动作仍是医生改标签。)

---

## 决策 8:班次=病人固定属性,所有透析日同一班,护士长/主班护士权限指定

**问题**:病人的班次(上午/下午/晚)怎么定?

**结论**:
- 班次是**病人的固定登记属性**;
- 同一病人**所有透析日都在同一个班**;
- **登记时人工指定**,系统只读取并据此排班(系统不自动挑班次);
- 指定有**权限限制:只有护士长和主班护士可以指定/修改班次**。

---

## ★ 架构提炼:病人"时间骨架" vs 排班算法职责

到决策 8 为止,病人的**时间骨架**已完全由人工权限属性确定:

| 维度 | 决定方式 | 权限 |
|------|----------|------|
| 分区(A/B/C) | 病人标签 | 医生/护士长(改标签) |
| 频率模式(哪几天) | 医嘱/登记属性 | 医嘱 |
| 班次(上午/下午/晚) | 登记属性 | 护士长/主班护士 |
| 治疗模式(HD/HDF/CRRT 及 HDF 日) | 医嘱属性 | 医嘱 |

→ **排班算法不决定上述任何一项,只读取。算法的唯一职责:在时间骨架(区+天+班+模式)约束下,为每一次透析分配具体的机器/床位。**

## 决策 9:新病人初始机位 = 三天同一台机优先 → 逐次分配 → 报警

**问题**:新病人(无历史机位)第一次怎么分配机器?

**结论**(策略 3,符合"主→备→报警"原则):
1. **首选**:找一台该病人**所有透析日都空闲的同一台机**,一次性固定下来(从第一次就实现固定机位)。
2. **退路**:找不到三天全空的同一台机 → **逐次分配**(各透析日分别找空位,可能不同机器)。
3. **报警**:仍排不开 → 报警人工裁决。

**含义**:
- 一旦初始机位确定,后续即转为"固定位置/就近"维持(决策见下一条)。
- 待确认:多台候选机时,挑机倾向**集中(空机连片)**还是**分散(负载均衡)**?(下一问)

---

## 决策 10:挑机偏好 = 集中(空机连片)+ 组团(同班次/同方案相邻)

**问题**:多台空机可选时挑哪台?

**结论**(A+C 结合):
- **集中**:优先让空机**连成片**,保护"整台全空机"这种稀缺资源(利于后续新病人找到三天全空的同一台)。
- **组团**:尽量把**同班次、同方案**的病人排在**相邻机位**,便于护士分组管理与交接班。
- **不做负载均衡**(刻意打散会破坏位置连续性,透析场景不适用)。

---

## 决策 11:模板复制生成;默认 2 周(可选 4 周);三级确认工作流

**问题**:未来 2~4 周排班怎么生成?周期多长?要不要草稿确认?

**结论**:

**① 生成方式 = 模板复制**。维护标准模板,生成时把稳定病人的模式原样往后铺;只有变动病人(新入/转出/改方案/请假)才单独处理。不做全量重排(保护位置稳定性、省算力)。

**② 周期 = 默认每 2 周;界面提供"4 周"选择按钮**(与 HDF "每两周一次"节奏对齐;4 周 = 2 个 HDF 周期)。

**③ 三级确认工作流**(逐级收紧,确保排班最终准确):

| 确认 | 时机 | 谁 | 对象 |
|------|------|-----|------|
| **第一次确认** | 草稿生成后 | 护士长(权限) | 整个 2/4 周排班草稿 → 生效 |
| **第二次确认** | 透析日**前一天 11:00 前**(周一的排班在**前一个周六**确认) | (主班/护士长) | 次日全天排班 |
| **第三次确认** | 当日**早上** | (主班/护士长) | 当日全天排班 |

**含义**:
- 状态机需支持多级确认:草稿 → 一次确认(计划生效) → 二次确认(次日确认) → 三次确认(当日确认) → 执行/透析中 → 完成。
- 第二次确认的"前一天"要处理**非工作日跳转**:周日不排,故周一的次日确认提前到周六。
- 三级确认之间允许"按现实情况微调"(请假、急诊加台等),每一级确认都是一次纠错关口。
- 待澄清:每一级确认之间具体允许改什么?(下一问可深入)

## 决策 12:临时透析 = 当前班次就近插空位(可借当天请假空位),不进模板,权限操作

**问题**:临时透析(第五种,计划外)怎么插进已排好的班?

**结论**:
- **落位**:在**当前班次、对应区**里就近找一台空闲机器插入(C 区急症/危重优先用 C 区空位);按其医嘱模式(HD/HDF/CRRT)匹配机型。当前班次无空位 → 看下一班次 → 仍无 → 报警人工。
- **可借用当天请假病人空出的机位**(当天一次性借用,**不改变**原病人的固定归属);但**不得挤走**已在排的规律病人。
- **不进模板**:临时透析是一次性的,只存在于当天实际排班,不参与往后复制。
- **权限操作**:医生开急诊透析医嘱 + 护士长/主班护士确认排入。

---

## 决策 13:不自动补透;系统只提示次数差异,补不补/补哪天由人工决定

**问题**:规律病人请假/缺席,这一次要不要补?补在哪?

**结论**:
- **提前请假**:系统把该次标记为**取消**;**是否补、补到哪天哪班由医生/护士长人工决定**并手工排入(系统给可用空位**建议**,不自动补)。
- **当天缺席(爽约)**:标记**缺席**状态(区别于正常取消,**留痕**);机位释放,可被临时透析借用(决策 12)。
- **系统不自动补透**,只**提示"本周该病人应排 X 次、已排 Y 次、差 Z 次"**(复用老系统差异检测),决定权交人。

**含义**:
- 状态需区分:正常取消 vs 缺席(爽约,留痕)。
- 差异检测(应排 vs 已排)是常驻提醒功能。

---

## 决策 14:方案变更带"生效日";生效日后未确认排班重排为草稿,已确认走人工

**问题**:医嘱/方案变更(频率/模式/班次/分区),已排的未来排班怎么改?

**结论**:
- **生效日机制**:医生改医嘱时指定**生效日**;生效日之前已排的不动,之后按新方案处理。默认生效日 = **次日**(或下一个未确认的排班日),由人确认。
- **按确认进度分界处理**:
  - 生效日后、**尚未二/三次确认**的排班 → 系统**按新方案重排为草稿**,重走确认流程。
  - 已临近、**已二/三次确认**的近期排班 → **不自动改**,需改则走人工调整。
- **机位联动**:改模式→触发 HDF 机位逻辑(决策 2/3);改分区→须医生改标签(决策 7);尽量保留原 HD 机位,只动变化部分。
- **高亮提醒**:方案一变,系统高亮"该病人 X 日起方案变更,N 条未来排班受影响,请审核"。

## 决策 15:默认严格不跨区;排满→报警进冲突队列→护士长特批,实时显示余位预警

**问题**:某区/某班排满怎么办?允不允许跨区?

**结论**:
- **默认严格不跨区**(A=门诊/B=住院 分区有管理意义)。
- 排满处理:排不进 → **报警,放入"待处理/冲突队列"** → 护士长权限裁决,选项包括:协调病人改班次/改日期、**启用加床**、或**特批跨区**。
- **跨区、改班、加床均为人工权限操作**,系统不自动执行。
- **容量模型**:每台机每班次 = 1 个透析位;区容量 = 区内机器数 × 班次数(+ 可加床位)。系统**实时显示每区每班余位**,将满时(余位 < 阈值)预警。

**含义**:数据模型需支持"加床"(临时增加透析位);需要"冲突/待处理队列"视图。

## 决策 16:统一排班状态机(模板独立存储;临时用来源标记;确认用时间戳)

**问题**:统一每条排班记录的状态机。

**结论**:

**① 模板独立存储**:模板**单独存表/单独标记**,不再用状态值兼任(根除老系统"删排班误删模板"的坑)。

**② 临时透析不设独立状态**:临时透析也是真透析,用 **`sourceType` 字段(常规/模板 vs 临时)** 区分,状态照常流转(草稿→确认→透析中→完成)。

**③ 三级确认用"已确认状态 + 三个时间戳字段"**:不设三个独立状态,状态机保持简洁;确认进度由时间戳记录。

**定稿状态机(每条排班记录)**:

| 编码 | 状态 | 含义 |
|------|------|------|
| 0 | 待排 | 应排未排(识别出来的缺口) |
| 10 | 草稿 | 模板复制/算法生成,未确认 |
| 20 | 已确认 | 计划生效。附带 3 个时间戳:`confirm1_at`(护士长一次确认)、`confirm2_at`(次日前 11:00 确认)、`confirm3_at`(当日确认) |
| 50 | 透析中 | 已上机 |
| 60 | 已完成 | 本次结束 |
| 70 | 已取消 | 提前请假/计划取消(留痕) |
| 80 | 缺席 | 当日爽约(留痕,机位可借) |

**正交字段(不进状态机)**:
- `sourceType`:常规(模板复制)/ 临时(急诊插入)
- `dialysisMode`:HD / HDF / CRRT(按次,决策 1)
- **"拒收"是病人登记环节的结果(病人级),不是排班记录状态**——拒收的病人根本不生成排班记录。

**含义**:数据模型 = 排班表(状态机如上)+ 独立模板表 + 病人属性(分区标签/频率/班次/模式/HDF日)+ 资源表(区/机器/床位,机型 HD/HDF/CRRT + 加床)。

# ▣ 阶段小结:16 条决策已覆盖的范围

- **HDF 策略**(决策 1–4):替换语义、双固定机位、固定 HDF 日+均匀性、两轮分配(HD 优先 HD 机)。
- **分区**(决策 5–7):标签驱动、门诊+传染病拒收、急诊人道进 C、不设传染病区、转区须医生改标签。
- **时间骨架**(决策 8):班次为权限登记的固定属性。
- **机位分配**(决策 9–10):新病人三天同机优先、集中+组团挑机。
- **生成与确认**(决策 11、16):模板复制、默认 2 周(可选 4 周)、三级确认、统一状态机。
- **扰动处理**(决策 12–15):临时透析插空位、不自动补透(只提差异)、方案变更带生效日、不跨区+排满报警+冲突队列。

---

## 决策 17:设备停机=权限登记时段→冻结机位→自动找临时替代位→修好归位

**问题**:某台机故障/维修停用,固定病人怎么办?

**结论**:
- **停机登记**:工程师/护士长权限把机器标记"停用"并填**停用时间段**;系统冻结该机在该时段的所有位。
- **临时停机(默认 ≤ 48 小时)**:系统**自动列出受影响排班**,为每个病人**就近找替代空位**(同区同班次最近、模式匹配)生成**调整草稿** → 护士长确认。**不改变原固定机位,修好后自动归位**。
- 找不到替代位 → 报警 → 冲突队列 → 护士长人工(改时间/加床/跨区特批)。
- **长期停用 / 报废(>48 小时)**:不自动定终身机位,**受影响病人由人工重新分配固定机位**(永久迁移)。

**含义**:数据模型需"机器停用时段"字段;迁移分"临时(可归位)"与"永久(重定)"两类。

## 决策 18:CRRT 纳入,作为 C 区特殊排班类型(不走三班/不走模板,只记机器+起止时间)

**问题**:CRRT 要不要纳入本排班子程序?

**结论**(方案 2):
- CRRT **纳入**本系统,作为 **C 区的一种特殊排班类型**。
- **不走三班网格、不走模板复制、不走固定机位逻辑**。
- 只记录:**占用了哪台 CRRT 机 + 治疗起止时间**(可跨班、跨天)。
- CRRT 机只能做 CRRT(机型约束);CRRT 病人属 C 区(危重)。
- 权限操作(医嘱 + 护士长/主班确认)。

**含义**:
- 排班记录需支持两种形态:**①规律透析(区+床/机+班+日期,走状态机/模板)**;**②CRRT(机+起止时间区间,特殊类型)**。
- CRRT 占用要在视图里显示,避免该 CRRT 机被误排。

## 决策 19:维护"非透析日"日历自动跳过;受影响病人挪班给建议草稿人工确认;支持假日值班模式

**问题**:法定假日整周/数日停透怎么处理?

**结论**:
- **非透析日日历**(权限维护):把不开诊的法定假日标入;系统生成排班时**自动跳过**(周日默认即非透析日)。
- **停掉的透析次**:系统**自动识别受影响病人**,生成"就近前后挪"的**建议调整草稿**(挪到假日前后可用空位)→ 护士长确认;**不自动定**(同补透逻辑,临床判断留人)。
- **假日值班模式**:假日可**只开放部分区/机器**(如只留 C 区 + 数台机)做急诊透析(临时透析,决策 12),非全院停机。

**含义**:数据模型需"机构日历(透析日/非透析日)"+"机器/区的可用日历(支持假日部分开放)"。

## 决策 20:容量模型 = 一班一台一人(不翻台);翻台靠"班次数量"表达

**问题**:一台机一个班次能否透多个病人?

**结论**(模型 A):
- **一个班次 × 一台机 = 1 个透析位**,班次内不再细分、不翻台。
- 一台机一天透三个病人,通过"上午/下午/晚"三个班次表达(三班本身就是翻台)。
- 若某机某天还要再加一轮 → 那是"新增一个班次"的问题,不在一个班内塞两人。

**含义**:
- 容量计算清晰:区容量 = 机器数 × 班次数(+ 加床)。
- 不需要按治疗时长/起止时间排队的复杂逻辑(CRRT 除外,决策 18)。

**核心三轴**:时间骨架(区+频率+班次,人工权限属性)× 机器能力(HD/HDF/CRRT)× 机位分配(固定+就近)。
**两条贯穿原则**:①主→备→报警人工;②算法只读骨架、只分机位。
**已覆盖**:HDF策略(1-4)、分区院感(5-7)、班次(8)、机位分配(9-10)、生成与三级确认+状态机(11/16)、扰动(12-15)、设备停机(17)、CRRT(18)、假日(19)、容量模型(20)。

# 设计阶段补充决策(D 系列,源自设计文档暴露的开放问题)

## 决策 21(D-6):奇偶周用"固定基准周一 + 连续周计数",根除漂移

**问题**:如何稳定定义"第几周/奇偶周"(服务 HDF 每两周一次的错峰)?

**结论**(方式 B):
- 设一个**全局固定基准周一**(配置项),默认 **2025-01-06(周一)**,可在配置中修改。
- 任意周序号 = (该周周一 − 基准周一) ÷ 7;**奇偶 = 序号 % 2**。连续计数,跨年无缝,永不漂移。
- 每个 HDF 病人有属性 `HDF周 ∈ {奇, 偶}`;系统在其入组时按当前两侧 HDF 机负载,分到人少的一侧(实现决策 3 的均匀性测算),之后固定每两周做一次。
- 新规则 5 种频率模式本身不用单双周;奇偶周**仅服务 HDF 错峰**。

**含义**:配置需"奇偶周基准周一"参数;`PatientProfile` 需 `HdfWeekParity` 字段。

## 决策 22(D-10):无"加床"模式;排满时到后面时段选空位机器安排

**问题**:排满时"加床"在"一班一台一人"模型下是什么?

**结论(修正决策 15)**:
- **系统中不存在"加床"概念**,不新增临时位/机动机。
- **排满时的处理 = 在后面的时段(后续班次 / 后续日期)里选择有空位的机器安排该病人。**
- 仍走"建议 + 护士长人工确认"(协调到可用的后续时段),无可用时段才报警进冲突队列。

**对其他决策/设计的修订**:
- **决策 15** 中"加床"作废;排满处理选项改为:协调改到后面时段(后续班次/日期的空位)、或特批跨区。
- **容量模型(决策 20)** 保持"机器数 × 班次数",**不含加床**。
- **设计文档** 中 Agent 2 建议的 `Schedule_ExtraSlot` / 机动机建模**取消**,无需该表/字段。

**含义**:溢出处理是"时间维度上顺延找空位",不是"空间维度上加位"。
