## 四、未采纳 —— 触及业务红线(重点)

### #21 / 3.1.7「一键挪位 / 自动解决冲突」——不予实现

本系统有一条**贯穿全部 22 条业务决策的铁律**(临床安全考量,由临床方明确要求):

> **主方案 → 备选方案 → 仍冲突则报警,交人工裁决。系统不自动解决冲突。**

"系统自动分析冲突并批量挪位"会让排班在**无人确认**的情况下被算法改动,违反该红线,临床上不可接受(排班涉及病人安全、家属接送、设备/院感约束,必须由护士长/主班护士拍板)。

**可以做、且符合规则的替代**:
- 系统**给出建议**(我们已有:冲突队列里带"建议挪到 X 机/日"),
- 由护士长**一键采纳某条建议**(人确认后才生效)。

即:**"系统建议 + 人工一键确认"可以;"系统全自动改排"不行。** 如需我们把"一键采纳建议"做出来,可在下一轮加(它仍是人工确认,不越红线)。

## 五、测试过程建议

1. **鉴权行为已变(重要)**:v1.1 默认安全。本地用网页测试时二选一:
   - 启动前 `set DEV_SUPERUSER=true`(网页默认不带角色也能操作),或
   - 在网页右上角「角色」选一个具体角色(默认已是"护士长")。
   - **生产环境不要设 `DEV_SUPERUSER`**,以保证未鉴权请求被拒。
2. **按权限矩阵测**:用不同 `X-Role` 验证 403:
   - 整盘确认=护士长;次/当日确认、取消/缺席/移床=护士长/主班;
   - 临时透析/CRRT=医生/护士长/主班;停机/假日=护士长;方案变更=医生/护士长;补透=护士长/主班。
3. **健康检查**:`GET /health` 应返回 `db:ok`;停掉数据库应返回 503。
4. **干净数据**:库被测乱后,`DROP DATABASE aihms; CREATE DATABASE aihms;` 重启后再点「写演示数据」。
5. **生成幂等**:对同一周重复点「生成排班」,第二次应 `drafts=0`(不重复、不复活已取消),这是预期。
6. **CRRT 时间**:按服务器本地时区解释,启动 DSN 建议带 `TimeZone=Asia/Shanghai`。
7. **发现"业务规则"层面的疑问,请反馈,不要自行改规则**:规则以 `透析排班子程序规则规范_v1.md` 为准。

## 六、红线汇总(任何改动都必须遵守)

1. **不自动解决冲突** —— 排不开一律报警,人工裁决(可"建议+人工采纳")。
2. **算法只读病人"时间骨架"**(分区/频率/班次/模式)—— 这些是医嘱/权限属性,程序不擅自更改。
3. **历史与治疗中排班锁定** —— 过去日期、已上机的排班不可改。
4. **模板独立存储** —— 绝不用排班状态值兼任模板(老系统的坑,已根除)。
5. **HDF 每两周替换一次** —— 替换某次 HD、且只能上 HDF 机;HDF 机优先留给 HDF 病人。
6. **权限矩阵** —— 确认、改排、医嘱、停机等按角色控制。
7. **病人主档** —— 当前是本地轻量表;接入正式环境须对接老库 `Register_PatientInfomation`,并替换为正式鉴权。
